# Gods Eye — Candidate Architectures & Adversarial Review
_Three independent architecture proposals, each stress-tested by an adversarial reviewer. The synthesis (ARCHITECTURE.md) merges the winner with every required change._

---

## TSDF-baseline-then-simplify, fully in idiomatic Rust. Six pipelined stages, one std::thread each, linked by bounded crossbeam channels (cap 2-3) that double as triple-buffers + backpressure: (1) capture (nokhwa) -> (2) metric depth (ort ONNX, DAv2-Metric-Small ViT-S, EP-selected behind a DepthBackend trait) -> (3) RGB-D-style direct VO from predicted depth (pure-Rust nalgebra SE3 + Gauss-Newton, keyframe pose-graph backend for drift) -> (4) sparse voxel-hashed TSDF integration in wgpu compute -> (5) marching cubes in wgpu compute over dirty blocks only -> (6) live viewer (rerun for dev, three-d+egui for shipped). Adaptivity (planar/QEM/progressive) is a deliberate POST-PROCESS on the extracted mesh: first get a correct live triangle mesh on screen on >1 platform (M0), then layer per-chunk QEM (meshopt LockBorder), RANSAC plane->2-tri quads, and observation-driven progressive refinement. All fusion+meshing stays pure wgpu/WGSL (no CUDA/ANE there) so there is ONE codebase and no perf cliff across Metal/Vulkan/DX12; optional CUDA/TensorRT/CoreML/DirectML live ONLY in the depth stage behind the trait, always with a CPU+wgpu fallback. Metric scale is owned by the learned depth model + an explicit intrinsics/scale-correction step, never by the VO; drift handled by keyframe pose-graph + (later) loop closure. Streaming host<->device block I/O is built from day one so discrete-VRAM boxes (Linux+NVIDIA/Windows) never cliff. Everything permissive (MIT/Apache/BSD); cargo-deny enforces no GPL/CC-BY-NC creep; no weights in the repo (first-run downloader).

> Pure-Rust monocular RGB -> ort/ONNX metric depth -> nalgebra RGB-D direct VO -> wgpu voxel-hashed TSDF -> wgpu marching cubes -> live mesh, with adaptive planar/QEM/progressive LOD as a post-process; portable wgpu+CPU baseline everywhere, optional CUDA/CoreML/DirectML behind a clean trait.

**Data flow:** Camera frame (nokhwa, YUYV/NV12 ~640x480) -> rayon/compute color-convert to RGB, recycled buffer from object pool → RGB frame -> resize to 392-518px (fast_image_resize SIMD) -> normalize (ndarray/half, F16) -> ort session.run() on selected EP -> metric depth map (HxW f32, meters) → Depth map + RGB + intrinsics -> scale/intrinsics correction (focal rescale, optional SLAM-recovered scale factor) -> metric depth → Corrected depth + previous keyframe -> RGB-D direct alignment (nalgebra SE3 Gauss-Newton, photometric+inverse-depth residual, robust Huber) -> 6-DoF pose delta -> pose-graph node; keyframe pose-graph optimized on a slow cadence for drift → Depth + pose -> wgpu: unproject to camera-space points (compute) -> integrate TSDF into hashed 8^3 blocks intersecting frustum near surface; mark touched blocks dirty; Welford TSDF variance per block accumulated → Dirty blocks -> wgpu marching-cubes 3-kernel pipeline (active-cell detect -> exclusive prefix-scan compaction -> vertex emit) -> per-block vertex/index deltas in GPU buffers → Mesh deltas -> small readback (final geometry only, never intermediate depth/TSDF) -> CPU mesh store → [POST-PROCESS, off hot path] per-chunk meshopt_simplify(LockBorder, ErrorAbsolute) QEM; RANSAC+region-grow plane detection -> 2-triangle quads for confirmed large planes; progressive refinement re-simplifies at monotonically shrinking target_error as per-region observation count rises → Final adaptive mesh -> double-buffered vertex/index buffers -> viewer stage (rerun log or three-d draw)

**Target fps:** Base M1 (8-core GPU, ort CoreML, portable wgpu fusion/mesh): honest ~18-25 fps end-to-end, throughput-bound by the depth stage (~40-55ms/frame DAv2-S F16 @392-518px); depth deliberately run at 15-20Hz with progressive multi-frame fusion filling detail, so the live mesh updates smoothly even though depth is sub-camera-rate. M1 number is INTERPOLATED (no published base-M1 DAv2 figure) — verify empirically before locking the budget. Mid Linux+NVIDIA (RTX-class, CUDA/TensorRT EP): depth ~13-20ms, so the pipeline becomes VO/fusion-bound and hits ~40-60 fps. CPU-only floor (tract/candle, no vendor EP): ~3-8 fps — usable for correctness/CI, not the live target.  
**Peak memory:** 1560 MB  
**Platforms:** macOS Apple Silicon (M1/M2/M3, arm64) — Metal via wgpu; depth via ort CoreML/ANE (CPU/XNNPACK fallback); primary dev/baseline box (macos-14 CI). ort 2.0 dropped x86_64 mac prebuilts, so Intel Macs use tract/candle CPU., Linux x86_64 + NVIDIA — Vulkan via wgpu; depth via ort CUDA/TensorRT EP (CPU fallback); needs libv4l-dev for nokhwa V4L2. Mid box (RTX-class) is the second reference target., Linux x86_64 + Intel/AMD GPU — Vulkan/GL via wgpu; depth via CPU/XNNPACK (no vendor EP)., Windows 10/11 x86_64 — DX12 via wgpu; depth via ort DirectML EP (CPU fallback); MSVC toolchain., Fallback floor everywhere: portable wgpu (or GL) compute + pure-Rust CPU depth (tract/candle) — runs with zero native ML toolchain., WebGPU/Wasm — viewer (rerun/egui) runs in-browser; full pipeline browser support is a stretch goal, not v1.

**Metric scale:** Scale is owned UPSTREAM by the learned metric depth, never by the VO — this is what turns the monocular problem into an RGB-D problem and kills classic scale ambiguity. Ship DAv2-Metric-Small (Apache-2.0) in TWO domain checkpoints, Indoor (Hypersim, 20m) and Outdoor (VKITTI, 80m), selected per deployment; DJI drone target uses Outdoor. CRITICAL: DAv2 metric output is SCALE-APPROXIMATE, not ground truth — it bakes in a ~470mm-class training focal and drifts ~10-20% when the real camera intrinsics differ. So ge-depth applies an explicit correction stage: (1) focal rescale using the actual camera intrinsics (known/calibrated, or from EXIF/telemetry; UniDepth-style intrinsics prediction kept ONLY in offline Python eval tooling); (2) an optional per-session global scale factor recovered from SLAM triangulation or a known-size reference, smoothed across keyframes. Per-frame depth-scale jitter would re-introduce scale drift, so the pose-graph backend enforces scale CONSISTENCY across keyframes (a single slowly-varying scale state, not per-frame). Because variance-adaptive LOD uses TSDF variance, and monocular depth is noisier than RGB-D, we subtract an expected depth-noise floor before using variance as the LOD criterion so noise does not masquerade as geometric detail (keeps flat walls coarse).

**Pose:** Native-Rust RGB-D-style direct visual odometry, no CUDA/GPL/FFI. Because each frame has metric depth, estimate frame-to-keyframe 6-DoF by direct alignment: minimize a robust (Huber) photometric residual plus an inverse-depth/geometric residual, solved with nalgebra SE3 Lie-algebra Gauss-Newton/LM (coarse-to-fine image pyramid via imageproc/image). glam on the hot inner loop. Keyframes spawned on translation/rotation/co-visibility thresholds. DRIFT (no IMU/LiDAR): a keyframe POSE-GRAPH backend (native LM, or optional cxx-FFI to BSD-licensed GTSAM kept behind the trait) optimized on a slow cadence; loop closure is first-class, not an afterthought — detect with a learned global descriptor + small Rust HNSW index (avoid C++ DBoW2), then add loop edges and re-optimize. Everything sits behind a PoseEstimator trait with impls: (a) native direct-metric VO [portable default, ships everywhere], (b) optional future wgpu patch-VO (DPVO-style corr+BA reimplemented in WGSL) as an accelerated path, (c) optional CUDA DPV-SLAM FFI (MIT) for NVIDIA boxes — never required. DPVO/DPV-SLAM (MIT) are the accuracy blueprint; ORB-SLAM3/DSO (GPLv3), DROID/MASt3R-SLAM (CUDA/>=11GB/NC weights) are design references only, never dependencies. Drone caveats (rolling shutter, autofocus-varying intrinsics, motion blur) handled by robust losses + online intrinsics/exposure handling.

**Fusion & meshing:** FUSION: voxel-hashing + TSDF in wgpu compute — the only substrate that makes BOTH cheap incremental updates AND adaptive resolution natural, and whose flat integer-hash + storage-buffer design ports cleanly to WGSL (no octree pointer-chasing). Spatial hash table of heap-allocated 8^3 voxel blocks storing TSDF+weight(+color); integrate a depth frame only into blocks intersecting the view frustum near the surface (truncated band), O(1) insert/lookup, fully GPU-parallel. Out-of-core streaming from day one: keep only blocks within a radius of the camera resident in GPU memory, evict the rest to host — mandatory for 8GB unified AND for discrete VRAM. Per-block Welford online TSDF variance accumulated for LOD. Room-scale storage is ~130-160MB (MrHash datapoint) — comfortably under 4GB. MESHING (baseline): GPU marching cubes ported from Will Usher's WebGPU 3-kernel pipeline (active-cell detect -> exclusive prefix-scan stream compaction -> vertex emit), run ONLY over dirty blocks each cadence — NEVER the full volume (a dense 256^3 is ~250ms on base M1; dirty-only keeps it <10ms). Double-buffered vertex/index buffers so the viewer never stalls. ADAPTIVE/PLANAR/PROGRESSIVE (post-process, the project differentiator): (1) drive LOD from the TSDF/voxel domain via the MrHash variance criterion — low-variance block => coarse => ~2 triangles for a wall, high-variance => fine; cleaner/more-incremental than mesh-domain decimation. (2) Tile the mesh per spatial-hash chunk and run meshopt_simplify with SimplifyLockBorder (crack-free, parallel per chunk via rayon) + ErrorAbsolute (metric, scale-stable LOD thresholds) + SimplifyVertex_Priority to protect plane-corner vertices. (3) CUSTOM RANSAC + normal-based region-growing (nalgebra SVD plane fit) on confirmed large planar regions -> emit a single 2-triangle quad, bypassing the voxel mesher (escapes the locked-border grid limit on wall-spanning chunks; hierarchically re-chunk as a region's confidence saturates). (4) PROGRESSIVE/observation-driven: per-region observation-count/confidence; re-invoke meshopt at a MONOTONICALLY shrinking target_error as evidence accumulates (coarse on first sight, finer over frames) — avoids writing a Hoppe vsplit half-edge engine; hysteresis + never-coarsen-a-stabilized-region prevents triangle popping. Crack-free across mixed resolutions uses Transvoxel transition-cell logic (reimplemented in the WGSL extractor) or neighbor-LOD-differ-by-at-most-one; same-resolution chunks use LockBorder. fast-surface-nets is the CPU fallback extractor for the no-GPU path.

**Performance plan:** Six std::thread stages joined by BOUNDED crossbeam-channels (cap 2-3): the bound gives backpressure AND acts as the triple-buffer (producer fills N+1 while consumer reads N). Preallocate ALL per-frame buffers (frame, depth tensor, point cloud, vertex/index pools) once and recycle via a return channel (object-pool pattern) so no stage ever stalls on the allocator and peak memory stays FLAT under the 8GB ceiling. rayon for embarrassingly-parallel inner loops (color-convert, per-chunk QEM, RANSAC). Depth dominates and is PIPELINED, so end-to-end fps tracks the depth stage, not the sum: depth runs at 15-20Hz (below camera framerate) and progressive multi-frame fusion integrates detail over time ('compute follows information'). GPU work: CHAIN every per-frame wgpu pass (unproject -> integrate -> [extract on cadence]) into a SINGLE command-encoder submission so intermediate depth/TSDF never leaves the GPU; only the ~1-6MB frame upload (write_texture) and the small final mesh-delta readback cross the bus. map_async ALWAYS paired with device.poll() (silent-hang trap). Budget a conservative 1.2-1.6x vs hand-tuned native for wgpu; use create_shader_module_unchecked in release to skip WGSL validation (a measured Metal regression). Mesh-extract runs on a separate cadence (every N frames / as budget allows), not every frame; the adaptive post-process runs off the hot path entirely. Fusion+extract are pure wgpu with NO optional-accel branch, so there is zero per-platform perf cliff there. PER-STAGE BUDGET (base M1, portable path, target ~18-25 fps / 40-55ms throughput): capture+colorconv 3ms; depth (DAv2-S F16 @392-518 ort+CoreML) 40-55ms (pipelined, throughput-bound); upload <1ms; VO 5-10ms; TSDF integrate 3-5ms; MC extract (dirty-only, cadence) 5-8ms; adaptive post-process 4-6ms (off hot path); viewer 4-6ms (own stage/process). Mid Linux+NVIDIA box (RTX-class, CUDA EP): depth ~13-20ms => the pipeline tracks VO+fusion and hits ~40-60 fps.

**Portability plan:** Two clean traits give optional-accel-with-fallback and no perf cliff. (1) DepthBackend trait: default ort impl that selects EP at RUNTIME — CoreML(ANE) on macOS, CUDA/TensorRT on NVIDIA, DirectML on Windows, CPU/XNNPACK everywhere as guaranteed floor (ort's built-in EP fallback chain IS the trait). Second impl: candle (pure-Rust, Metal/CUDA/CPU) for no-C++-runtime environments. Third: tract (pure-Rust CPU) as absolute no-native-toolchain floor. (2) PoseEstimator + ComputeBackend traits: native-Rust direct VO + wgpu fusion/mesh are the ALWAYS-AVAILABLE default; optional wgpu patch-VO and optional CUDA DPV-SLAM FFI sit behind the trait as accelerated paths, never required. Accel backends are cargo FEATURES inside the backend crates (coreml/cuda/tensorrt/directml/wgpu), so default `cargo build` compiles a portable CPU+wgpu path on every OS with no combinatorial release churn. ALL fusion/meshing/viewer GPU work is one WGSL codebase (Naga -> MSL/SPIR-V/HLSL/GLSL) — Metal/Vulkan/DX12/GL from one source; subgroups/mesh-shaders kept behind feature flags, never baseline (GL/older-backend portability). Do NOT architect around unified memory: explicit staging buffers + host<->device block streaming from day one so discrete-VRAM Linux+NVIDIA/Windows have no cliff (only frame upload + mesh readback cross PCIe). Pin EXACT versions of wgpu and ort in [workspace.dependencies] (both fast-moving; ort still rc). PACKAGING: no weights in repo — first-run downloader (hf-hub + sha256 + directories-next cache) with per-model license-acceptance + --model-path offline flag for drone. CI matrix ubuntu(+libv4l-dev)/windows(MSVC)/macos-14(arm64), default job pure cargo (clippy/test/cargo-deny), accel EP builds isolated to opt-in jobs where C/cmake may intrude.

**Rust stack:** Inference: ort 2.x (ONNX Runtime bindings, download-binary, EP features) PRIMARY; candle+candle-transformers (pure-Rust) FALLBACK; tract (pure-Rust CPU) FLOOR. GPU-compute: wgpu 29 + WGSL + naga (portable Metal/Vulkan/DX12/GL/WebGPU). Camera: nokhwa 0.10 (input-native); image + fast_image_resize; openh264-rs/retina for later drone H.264. Math: glam (hot path SIMD) + nalgebra (SE3/SO3, SVD, Gauss-Newton/LM); ndarray + half (tensor pre/post). Mesh: meshopt (FFI to meshoptimizer >=1.0 via cc crate, no-cmake) for QEM; fast-surface-nets (CPU isosurface fallback); transvoxel (LOD-seam reference); ilattice/ndshape (chunk math); block-mesh (greedy coarse-quad option). Concurrency: std::thread + crossbeam-channel (bounded) + rayon. Viewer: rerun/re_sdk (dev) + three-d + egui/egui-wgpu (shipped). Packaging: hf-hub + directories-next + sha2 (weights); cargo-deny (license gate); pyo3+maturin (OPTIONAL offline tooling only). Optional FFI: cxx (only if wrapping BSD GTSAM pose-graph or MIT DPV-SLAM CUDA — never GPLv3).

**Immature spots:**
- MONOCULAR VO/SLAM has NO production-ready Rust crate (kornia-slam is GSoC-2026 roadmap, not implemented). This is THE biggest gap — the entire ge-slam stage (direct alignment, keyframe pose-graph, loop closure) is hand-written on nalgebra/glam primitives. No FFI needed (depth supplies scale), but budget the most engineering here. Rust-CV crates (arrsac/essential-matrix/PnP) are semi-maintained — vet before depending.
- GPU TSDF fusion AND GPU/wgpu marching cubes have NO mature crate — all WGSL kernels are hand-authored (port Will Usher's MC pipeline). The CPU mesher crates (fast-surface-nets/transvoxel/block-mesh) cannot carry the real-time GPU path. threecrate advertises wgpu-TSDF+MC+QEM but is low-traction/unproven — mine it for the shader approach, do NOT depend on it.
- WGSL exclusive prefix-scan (stream compaction) is the one nontrivial GPU primitive with no wgpu built-in — implement it yourself; atomics/subgroup support varies across Metal/Vulkan/DX12/GL, so test the GL/older-backend fallback or risk a portability cliff.
- RANSAC/region-growing PLANE detection and Hoppe progressive-mesh (vsplit/ecol log) have NO mature Rust crate — hand-written (~few hundred lines for RANSAC via nalgebra SVD; progressive done pragmatically as re-simplify-at-shrinking-target_error to avoid a half-edge engine).
- ort CoreML-without-cmake is UNVERIFIED (Microsoft does not ship prebuilt CoreML/DirectML; sources conflict on whether ort 2.x rc bundles CoreML or triggers a from-source cmake build). SPIKE-verify on the M1 box before committing CoreML; CPU/XNNPACK is the guaranteed no-cmake floor and tract is the pure-Rust backstop.
- Metric scale from one camera is not crate-solvable: DAv2 metric is domain-specific + focal-biased (10-20% off), so the explicit intrinsics/scale-correction + cross-keyframe scale-consistency code is custom. candle's depth_anything_v2 example targets the RELATIVE model, so a candle metric path means porting the metric head yourself (ort consumes official ONNX directly with less work).
- Crack-free mixed-resolution seams (Transvoxel transition cells) must be reimplemented in WGSL — the transvoxel crate is CPU/experimental and re-extracts on LOD change, unusable at runtime.

**Milestones:**
- **M0** (15-25): Smallest loop that puts a LIVE triangle mesh on screen and builds on >1 platform via cargo. Prove the spine end-to-end. → cargo run streams webcam (nokhwa) -> ort DAv2-Metric-Small ONNX depth (CPU/XNNPACK floor, EP optional) -> unproject -> single fixed-resolution wgpu voxel-hashed TSDF integrate -> wgpu marching cubes over dirty blocks -> rerun live mesh. Identity/simple VO first. Builds + runs on macOS(M1,Metal) AND Linux(Vulkan) in CI; default cargo build is pure CPU+wgpu, no cmake, no weights in repo (first-run downloader).
- **M1** (20-30): Metric pose + drift control: real RGB-D direct VO with metric scale, keyframe pose-graph. → ge-slam direct alignment (nalgebra SE3 GN, coarse-to-fine) replaces identity pose; intrinsics/scale-correction stage; keyframe pose-graph backend; trajectory + mesh consistent over a room loop. DepthBackend trait selects CoreML/CUDA/DirectML EP at runtime with CPU fallback.
- **M2** (18-28): Adaptive/planar mesh: wall -> 2 triangles, complex -> many. → ge-mesh-adapt: per-chunk meshopt QEM (LockBorder+ErrorAbsolute), RANSAC+region-grow plane detection emitting 2-tri quads, TSDF-variance-driven LOD with depth-noise-floor subtraction. Crack-free seams (LockBorder same-res, Transvoxel mixed-res).
- **M3** (20-30): Progressive observation-driven refinement + loop closure. → Per-region observation-count -> monotonic shrinking target_error re-simplify (coarse-on-arrival, finer over frames, hysteresis no-popping); loop-closure detection (learned descriptor + HNSW) + pose-graph re-optimization; out-of-core block streaming validated on discrete-VRAM box.
- **M4** (20-30): Portability hardening + packaging + perf. → CI matrix green ubuntu/windows/macos-14; accel EPs (CUDA/TensorRT/DirectML/CoreML) verified in opt-in jobs; cargo-deny passing; three-d+egui shipped standalone viewer; timestamp-query per-pass GPU profiling; create_shader_module_unchecked release path; documented per-platform fps. Optional drone H.264 source + outdoor checkpoint.

**Top risks:**
- Base-M1 depth latency is INTERPOLATED (no published figure) and vendor depth benchmarks are unreliable (DAv2-S claimed 13ms but measured ~98ms on some GPUs) — the whole fps budget rests on an unverified number. MITIGATE: spike-benchmark ort CoreML/CUDA EP on real target hardware in M0 before locking the budget; run depth at lower res (392) and sub-camera-rate.
- ort CoreML/DirectML may trigger a from-source cmake build (Microsoft ships no prebuilt CoreML/DirectML), breaking the no-cmake guarantee. MITIGATE: ship CPU/XNNPACK as the guaranteed no-cmake floor, keep tract/candle pure-Rust fallbacks, make CoreML an opt-in feature, spike-verify early.
- Largest engineering surface (VO/SLAM, GPU TSDF, GPU marching cubes, RANSAC planes, progressive mesh) is ALL greenfield Rust with no mature crate — schedule risk is real. MITIGATE: M0 ships fixed-resolution + identity-then-simple VO to de-risk the spine before adaptivity; reuse permissive blueprints (Open3D/RTAB-Map/DPVO algorithms).
- Monocular depth-scale jitter between frames re-introduces scale drift even though depth supplies absolute scale. MITIGATE: single slowly-varying scale state enforced across keyframes in the pose-graph; never trust per-frame scale.
- Dirty-set explosion: drift/loop-closure re-touching old blocks can blow up the per-frame extract cost and latency spikes. MITIGATE: cap dirty-block budget per cadence, prioritize near-camera blocks, amortize re-mesh over frames.
- Discrete-GPU perf cliff if the design leans on M1 unified memory zero-copy. MITIGATE: explicit staging buffers + host<->device block streaming from day one, only frame upload + mesh readback cross PCIe, validate on NVIDIA box in M3.
- WGSL prefix-scan + atomics/subgroup support varies across Metal/Vulkan/DX12/GL — a portability cliff on older backends. MITIGATE: baseline avoids subgroups, test GL fallback in CI, keep subgroup fast-path behind a feature flag.
- meshopt FFI / cc-crate build needs clang/gcc/MSVC present (not zero-toolchain) and the binding must vendor meshoptimizer >=1.0 for v1.1 flags. MITIGATE: pin the submodule, document the C-compiler requirement, pure-Rust meshopt port as last-resort fallback.

**Why this wins:** It is the lowest-risk path to a LIVE mesh that still satisfies every hard constraint. The TSDF-then-simplify spine is proven (Open3D-class algorithms) so M0 puts triangles on screen on >1 platform fast, with adaptivity layered on afterward rather than gating first light. Scale is solved upstream by permissive Apache-2.0 DAv2-Metric-Small, turning the monocular problem into a tractable RGB-D one and letting a pure-Rust VO own pose without CUDA/GPL/FFI baggage — the only pose path that runs on 8GB M1 AND transfers cleanly to a drone. Performance and portability are co-designed, not bolted on: depth accel (CoreML/CUDA/DirectML) is the ONLY place with optional backends, all behind one trait with a guaranteed CPU floor, while every fusion/mesh/viewer kernel is a single portable WGSL/wgpu codebase with no per-platform cliff and explicit host<->device streaming so discrete GPUs don't fall off. Memory stays flat and well under 4GB via sparse voxel hashing, recycled buffer pools, and dirty-only extraction. The licensing is airtight (dual MIT/Apache, cargo-deny gate, no weights in repo, every blocker documented), and the adaptive variance-driven planar/progressive mesh is the genuine differentiator — wall to 2 triangles, detail accruing where information accrues — built on the mature load-bearing crates (meshopt, fast-surface-nets, nalgebra) with the known immature pieces (VO, GPU TSDF/MC) explicitly scoped as the engineering budget.

### ⚖️ Adversarial verdict: **keep-with-changes**
- realtime feasible: False | fits modest memory: True
- **fps reality:** The 18-25 fps claim is the optimistic edge of an unverified budget, and the depth latency that the whole budget hinges on is roughly 2x worse than assumed. Measured DAv2-Small ONNX runs ~99-100ms at 518px (on hardware at least as capable as a base M1 GPU); the proposal budgets 40-55ms. Even pinned to 392px (which trades scale accuracy — see scale critique), a base-M1 8-GPU-core box with CoreML/ANE realistically lands depth at ~55-90ms, i.e. throughput ~11-18 fps, NOT 18-25. The proposal's own top-risk note admits the M1 number is interpolated; the honest pre-spike planning number is sub-15 fps end-to-end, with the live mesh updating at depth cadence (which the design explicitly accepts via multi-frame fusion). Two more under-budgeted items: (1) the WGSL exclusive prefix-scan in marching cubes is the dominant cost in dirty-block meshing and is highly backend-sensitive — 5-8ms is plausible on Metal but can blow out on GL/older Vulkan; (2) the VO is hand-written direct alignment with coarse-to-fine pyramids — 5-10ms is achievable only after real optimization, and is the first thing that regresses on a dirty-set spike. Net: treat 'real-time' as 'live-at-depth-cadence, ~10-15 fps on base M1' and require an empirical M0 spike before any number is locked.
- **memory reality:** The 1560MB peak is plausible and fits alongside macOS's ~3-4GB in 8GB — sparse voxel hashing (130-160MB room-scale datapoint is credible), recycled buffer pools, and dirty-only extraction keep it flat. But the stated total is a steady-state best case and omits three things that bite on the exact baseline box: (1) ort's ONNX Runtime + CoreML graph compilation and the ANE model partition hold transient working set well above the 260MB listed during first inference and on every input-shape change — budget a 400-500MB depth-stage spike. (2) rerun's in-process dev viewer accumulates a recording buffer; the proposal correctly notes memory_limit, but on 8GB the dev loop will OOM-thrash unless that limit is set aggressively low from day one, and rerun-in-process competes with the core for unified memory. (3) wgpu staging buffers for host<->device streaming are double-counted optimistically — on a discrete box the same blocks live in both host and VRAM during eviction. Verdict: fits, but the real ceiling under macOS is ~4GB of headroom, so 1560MB is comfortable ONLY if the dev viewer is moved out-of-process (rerun's gRPC mode) early; the in-process default will not be comfortable.
- **scale/drift:** This is the load-bearing weakness and the proposal both correctly identifies it AND understates it. 'Scale is owned upstream by the learned depth, turning monocular into RGB-D' is the right framing but the depth is NOT metric-faithful: DAv2-Metric is domain-and-focal-biased (the proposal admits 10-20%), and critically its per-frame depth is internally inconsistent — relative depth ordering is good, absolute scale wanders with content, exposure, and texture. Feeding that into a 'direct RGB-D VO' means the photometric+inverse-depth residual is minimized against a depth map whose scale silently breathes frame-to-frame. The proposed fix — 'a single slowly-varying scale state enforced across keyframes in the pose-graph' — is a real technique but it is NOT a solved crate; it is bespoke estimation that couples depth-scale, pose, and the pose-graph, and getting it stable over a multi-minute session is research-grade, not engineering-grade. Concrete failure modes the design will hit: (a) on a long wall pan the metric depth's scale drifts smoothly, the VO has no parallax to contradict it (degenerate forward/rotational motion), and the mesh inflates/deflates — classic monocular swim; (b) low-texture indoor scenes (white walls — exactly the '2-triangle wall' showcase) starve the photometric residual, so pose is depth-driven and drifts; (c) without IMU, scale is observable ONLY from translational parallax, which a hovering drone or slow pan does not provide. Loop closure helps global consistency but cannot fix locally-wrong metric scale. The drone target makes this worse: rolling shutter + autofocus-varying intrinsics break the fixed-focal assumption the scale correction depends on. Bottom line: the mesh WILL warp/swim on degenerate motion and texture-poor scenes, and the proposal's mitigations are the right list but are scoped as if they're tractable add-ons rather than the central research risk. This stage, not the GPU work, is what determines whether the project succeeds.
- **mesh quality:** The adaptive/planar/progressive scheme is the genuine differentiator but is also where cracks and flicker are most likely, and the proposal hand-waves the hardest seam. Same-resolution chunks via meshopt SimplifyLockBorder are genuinely crack-free — that part is sound and meshopt is mature. The problems: (1) mixed-resolution seams. The plan relies on 'Transvoxel transition cells reimplemented in WGSL' OR 'neighbor-LOD-differ-by-at-most-one'. The proposal itself flags transvoxel as CPU/experimental and that a WGSL reimplementation is required — this is a substantial, under-budgeted piece, and Transvoxel is patent-historically murky and notoriously fiddly. Until it exists, any RANSAC-plane 2-tri quad abutting a fine voxel-meshed region is a guaranteed crack, because the plane quad bypasses the voxel grid entirely (the proposal even says it 'escapes the locked-border grid limit') — so its border does NOT match the neighbor's locked border. That is a direct contradiction: you cannot both lock borders for crack-freeness AND bypass the grid with plane quads at the same boundary. (2) Popping/flicker. 'Monotonic shrinking target_error + hysteresis + never-coarsen-a-stabilized-region' is a reasonable heuristic but progressive re-simplification re-runs QEM and will pop vertices when a region crosses an observation threshold; without temporal vertex-morphing (geomorphing) there will be visible snaps, especially as the RANSAC plane detector flips a wall between 'voxel-meshed' and '2-tri quad'. (3) TSDF-variance-driven LOD with monocular depth: the noise floor subtraction is a good instinct but monocular depth noise is structured (worse at edges/low-texture), not white, so a single scalar floor will both leave noise-driven detail on textureless walls and over-smooth genuine fine geometry. Net: crack-free is achievable for the pure-voxel path; the plane-quad/voxel boundary is an unsolved seam the proposal contradicts itself on, and progressive transitions will flicker without geomorphing that isn't scoped.
- **portability:** Mostly genuinely cross-platform, with two honest caveats and one over-claim. Genuinely portable: the all-WGSL fusion/mesh/viewer path through Naga is the right call and avoids a CUDA lock-in — this is the proposal's strongest design decision. The optional-accel-behind-a-trait pattern with a CPU/wgpu floor is correct and means default `cargo build` runs everywhere. License story is good in shape: dual MIT/Apache, cargo-deny gate, no weights in repo, and the blocker list (ORB-SLAM3/DSO GPLv3 viral-even-via-FFI, UniDepth/Metric3D CC-BY-NC, MASt3R-SLAM NC, OpenMVS AGPL, DAv2 Base/Large/Giant + all metric tiers above Small CC-BY-NC) is accurate and is the right set to exclude. The over-claim: 'licensing is airtight' and 'Apache-2.0 on BOTH code and Small weights'. The DAv2-Small *metric* checkpoints inherit a real, unresolved dataset-license question — they were fine-tuned on Hypersim (which carries a research/EULA-style data license) and Virtual KITTI; there is an open upstream issue specifically questioning whether the Apache-2.0 weight label survives the training-data licenses for commercial use. This is NOT airtight; it's 'permissive label, unverified provenance.' Caveat 2: CoreML is actually IN the ort/pyke prebuilt binaries for Apple targets (so the proposal's 'CoreML may trigger cmake' risk is partly overblown for CoreML specifically) — but CUDA/TensorRT/DirectML EP prebuilts are version-pinned and the wrong combination DOES fall back to a from-source build that pulls cmake, so the no-cmake guarantee holds ONLY for the default CPU/XNNPACK + CoreML path, exactly as the floor is specified. Net: cross-platform yes; CUDA-locked no; but 'airtight licensing' must be downgraded to 'permissive with one unresolved weight-provenance question that needs legal sign-off before any commercial claim.'
- **Rust maturity:** The proposal's own immature-spots list is unusually honest and largely correct, which is to its credit — but it underweights how much of the project is concentrated in those gaps. Load-bearing mature crates that will hold: wgpu 29 (production-grade, Firefox/Bevy-proven), nalgebra/glam, rayon, crossbeam, meshopt (FFI, mature, but needs a C compiler — present here as clang, fine), image/fast_image_resize, hf-hub. ort is production-usable but still rc/fast-moving — pinning is mandatory and EP behavior shifts between releases. Where Rust genuinely bites, in order of pain: (1) There is NO production monocular VO/SLAM crate in Rust — kornia-slam is a 2026 roadmap item, Rust-CV (arrsac/essential-matrix/PnP) is semi-maintained. The ENTIRE ge-slam stage (direct alignment, Lie-algebra GN/LM, keyframe pose-graph, loop closure with HNSW) is greenfield. This is the single biggest schedule and correctness risk and it sits exactly on top of the scale/drift problem above — the two hardest problems are the same unbuilt stage. (2) GPU TSDF and GPU marching cubes have no mature crate; all WGSL is hand-authored, including the prefix-scan stream-compaction primitive whose atomics/subgroup behavior varies across Metal/Vulkan/DX12/GL — a real portability cliff on GL/older backends, correctly flagged. (3) Transvoxel-in-WGSL and RANSAC-plane and progressive-mesh are all hand-written. nokhwa is the quietly risky 'mature' dependency: it has documented AVFoundation frame-capture and NV12->RGB conversion bugs on M1 (the exact baseline box) — the '3ms, no accel needed' capture stage is real but flaky, and a capture-layer bug will look like a pipeline bug. Verdict on maturity: the design correctly isolates the immature pieces, but they are not peripheral — VO/SLAM + GPU TSDF/MC + adaptive mesher is ~70% of the novel engineering and ALL of it is greenfield. The Rust path adds no FFI for the depth model (good) but offers zero head-start on the hardest stage.

## PLANAR-ADAPTIVE-FIRST in pure Rust: never build a dense uniform TSDF. Each frame produces metric depth (DAV2-Metric-Small via ort) + a 6-DoF pose (native RGB-D-style direct VO that uses the metric depth to kill monocular scale ambiguity). The fused representation is a sparse spatial-hash of surface "cells" plus an explicit plane registry. Big flat surfaces are detected immediately by RANSAC + normal region-growing on the back-projected metric depth and represented as 2-triangle quads from frame one; non-planar regions keep only sparse detail patches that subdivide progressively as observation count and geometric-residual evidence accumulate ("compute follows information"). Drift is fought with depth-supplied scale consistency across keyframes + a native pose-graph backend with bag-of-words loop closure. Everything portable runs on CPU + wgpu; CUDA/CoreML/DirectML are optional accel ONLY for the depth model, behind a trait, with a guaranteed CPU/wgpu floor. This is the most memory-frugal and most CPU-algorithmic of the candidate strategies (least wgpu-kernel surface area), truest to the wall=2-triangles vision, at the cost of higher mesh-consistency risk which we mitigate with locked-border chunking, plane snapping, and monotonic refinement hysteresis.

> A pure-Rust, planar-adaptive monocular-to-mesh pipeline: DAV2-Metric-Small (ort, Apache-2.0) for metric depth, native depth-assisted direct VO for pose, an immediate-plane + progressive-patch sparse mesh, all portable on CPU+wgpu with optional CoreML/CUDA/DirectML accel behind a trait.

**Data flow:** Capture: nokhwa grabs a camera frame (V4L2/AVFoundation/MediaFoundation), YUYV/NV12 -> RGB via rayon or a small compute pass; preallocated frame buffer pulled from a return-channel pool. → Resize: fast_image_resize downsamples to the depth model's input (392px default, 518px quality mode), F16 normalized tensor (ndarray + half). → Depth: DepthBackend (ort, EP auto-selected) infers DAV2-Metric-Small -> metric depth map (meters). Runs on its own thread BELOW camera fps (~12-18Hz); scale-correction stage applies intrinsics + per-keyframe scale factor. → Unproject: depth + intrinsics -> metric point cloud + per-point normals (wgpu compute on portable path, or rayon CPU). Stays GPU-resident where possible. → Pose (VO): PoseEstimator aligns current frame's points/intensities to the active keyframe model by direct photometric + inverse-depth Gauss-Newton (nalgebra SE3); emits metric 6-DoF. Keyframe spawned on motion/parallax threshold. → Fusion: points routed by a spatial hash. RANSAC + normal region-growing assigns inliers to existing/new planes (plane registry); residual points land in sparse non-planar detail cells with running mean/variance (Welford) and observation counts. Only frustum-near cells touched. → Mesh build: planes emitted as 2-triangle quads (convex-hull / oriented bounding rectangle in plane basis); dirty non-planar cells meshed with fast-surface-nets then meshopt_simplify (LockBorder, ErrorAbsolute) to an observation-driven target_error. Double-buffered vertex/index deltas. → Viewer: rerun (dev) or three-d+egui (shipped) consumes mesh deltas + trajectory over a bounded channel; runs in its own stage/process so it never stalls the core. → Backend (drift): keyframe pose-graph + BoW loop closure runs at low cadence on a side thread; loop corrections re-anchor plane registry and trigger localized re-mesh of affected chunks.

**Target fps:** Base M1 (8 GB, 392px F16 depth via CoreML EP, portable wgpu fusion/mesh): ~16-22 fps end-to-end, depth-stage-bound (~45-55ms depth pipelined; M1 number is INTERPOLATED from M3 Max ~25ms / iPhone ~34ms and must be verified empirically). Mid Linux+NVIDIA box (e.g. RTX 3060/4070, CUDA/TensorRT EP, 518px): ~35-45 fps, with headroom; depth ~13-20ms. CPU-only floor (tract): ~3-6 fps, usable for headless/CI only. Depth deliberately runs below camera fps and the progressive multi-frame fusion integrates detail over time, so the system stays responsive even when single-frame depth is slow.  
**Peak memory:** 1590 MB  
**Platforms:** macOS Apple Silicon (M1+): Metal via wgpu, CoreML/ANE via ort — baseline dev box, Linux x86_64 + NVIDIA: Vulkan via wgpu, CUDA/TensorRT via ort, Linux x86_64 + AMD/Intel GPU: Vulkan via wgpu, CPU/XNNPACK depth, Windows 10/11 x86_64: DX12 via wgpu, DirectML via ort, Portable CPU-only floor everywhere (tract depth + CPU fusion/mesh; wgpu-GL fallback for old GPUs), Intel macOS: CPU/tract only (ort 2.0 dropped x86_64 mac prebuilts, min macOS 13.4) — documented degraded path

**Metric scale:** Scale is owned by the learned metric model, NOT the VO. Ship DAV2-Metric-Small (Apache-2.0) in two checkpoints: Indoor (Hypersim, 20m) and Outdoor (VKITTI, 80m), selected per deployment; the DJI drone uses Outdoor. Because DAV2 metric bakes in a ~470mm-class training focal length, raw output is treated as SCALE-APPROXIMATE (~10-20% off when intrinsics differ), so a mandatory scale-correction stage runs: (1) apply known/estimated camera intrinsics to re-project (corrects the focal-length bias), and (2) maintain a slowly-adapted global scale factor s estimated by aligning VO-triangulated sparse-feature depths against the model's predicted depths at keyframes (median-ratio robust estimate), clamped and low-pass filtered to avoid per-frame scale jitter. The backend additionally enforces inter-keyframe scale consistency so per-frame depth-model drift does not re-introduce scale drift. Metric3D-V2 is intentionally NOT shipped (BSD "non-commercial" wording is a flagged ambiguity); DAV2-Metric-Small is the clean Apache-2.0 metric choice. For unknown-intrinsics drone footage, offline Python tooling (UniDepth, research-only, never shipped) can be used to calibrate/validate intrinsics priors.

**Pose:** Native-Rust depth-assisted direct visual odometry. The metric depth turns monocular into RGB-D, so pose is a frame-to-keyframe SE3 estimated by minimizing a robust (Huber) photometric + inverse-depth geometric residual via Gauss-Newton/LM in nalgebra, on a 3-level image pyramid (coarse-to-fine) built with imageproc. glam carries the per-pixel hot loop; rayon parallelizes residual/Jacobian accumulation. A sparse ORB/FAST + PnP-RANSAC (arrsac) path provides robust re-initialization on tracking loss and large motion. Keyframes spawn on translation/rotation/parallax thresholds. This avoids all license/CUDA/FFI baggage of ORB-SLAM3 (GPLv3), DROID/DPVO (CUDA kernels), MASt3R (CC-BY-NC weights). The optional high-accuracy mode mirrors the MIT DPV-SLAM design (deep patch front-end + classical pose-graph backend) behind the PoseEstimator trait, with its cuda_corr/cuda_ba kernels reimplemented in wgpu — gated, never the default. The native direct VO is the always-available portable floor.

**Fusion & meshing:** No dense TSDF. A spatial hash (8^3-style cells, integer-keyed flat buffers) routes incoming metric points. PLANE PATH: each keyframe runs RANSAC plane proposals seeded from cell normals, then normal-based region growing across cells; an accepted plane is stored in a plane registry (centroid, normal, in-plane 2D extent as an oriented bounding rectangle / convex hull, supporting-observation count). A registered plane is emitted as exactly 2 triangles (the quad of its in-plane extent); growing extent just enlarges the quad, so a whole wall stays 2 triangles regardless of size. Inlier points are removed from the non-planar pool, which is the key memory win. NON-PLANAR PATH: residual cells keep running mean position + normal + Welford TSDF-like variance + observation count; cells are marked dirty on update. PROGRESSIVE REFINEMENT: a separate cadence meshes only dirty non-planar regions with fast-surface-nets, then meshopt_simplify with SimplifyLockBorder (chunk-independent, watertight borders) and SimplifyErrorAbsolute (world-unit, scale-stable LOD thresholds). target_error shrinks MONOTONICALLY as observation count crosses thresholds (coarse on first sight, finer over frames) with hysteresis so triangles never pop/coarsen — directly the "detail added progressively over multiple frames" requirement. CRACK SAFETY: same-resolution chunk seams handled by LockBorder; plane/non-planar boundaries snapped to the plane; transvoxel transition-cell logic is the reference if mixed-resolution seams ever appear. Because depth is noisy, an expected depth-noise floor is subtracted from cell variance before it drives subdivision, so flat walls do not wrongly stay high-res. Loop closures re-anchor plane poses and trigger localized re-mesh of only affected chunks (bounded dirty set is critical — unbounded re-touch is the latency risk).

**Performance plan:** Per-stage std::thread pipeline joined by bounded crossbeam-channels (capacity 2-3) that simultaneously give backpressure and act as triple-buffers: capture -> depth -> VO -> fusion -> mesh -> viewer, plus a low-cadence backend thread. All per-frame buffers (frame, depth tensor, point cloud, vertex/index pools) are preallocated once and recycled via return-channels (object-pool) so no stage stalls on the allocator and peak memory stays flat under the 8GB ceiling. rayon parallelizes inner loops (color-convert, residual/Jacobian accumulation, per-chunk surface-nets + meshopt). Depth runs on a dedicated thread BELOW camera fps (~12-18Hz) and is throughput-pipelined, so end-to-end fps tracks the depth stage while VO/fusion/mesh interleave. Latency budget (base M1, portable path, 392px F16 depth via CoreML EP): capture+convert ~3ms, resize ~1ms, depth ~45-55ms (pipelined, the bottleneck), unproject ~2ms, VO ~8ms, fusion ~6ms, mesh (dirty-only, amortized) ~7ms, upload/readback <1ms -> ~16-22 fps effective. Mid Linux+NVIDIA box (CUDA/TensorRT EP, 518px): depth ~13-20ms, other stages similar or faster -> ~35-45 fps, with the only PCIe traffic being a ~1-6MB frame upload + small mesh-delta readback per frame (all compute passes chained into one command-encoder submission; map_async always followed by device.poll). wgpu compute budgeted at a conservative 1.2-1.6x vs native; create_shader_module_unchecked in release to skip WGSL validation. Optional-accel split: only ge-depth uses CoreML/CUDA/DirectML; fusion/mesh/viewer are pure wgpu/CPU identically on every platform.

**Portability plan:** Two clean traits drive optional-accel-with-fallback. DepthBackend: impls are OrtBackend (default; registers EPs in priority order [CoreML, CUDA/TensorRT, DirectML] then CPU — ort's own EP fallback chain), CandleBackend (pure-Rust Metal/CUDA/CPU, no C++ runtime, honors no-cmake), TractBackend (pure-Rust CPU-only floor). ComputeBackend / PoseEstimator: portable wgpu/CPU default everywhere; an optional CUDA DPV-SLAM FFI and optional wgpu-accelerated patch VO sit behind the trait for accel boxes but are never required. Default `cargo build` on any OS compiles ONLY the portable CPU + wgpu + ort-download-binary + nokhwa path — no cmake, clang-only (clang ships with Metal toolchain on M1, present on Linux/Windows). Accel EPs (coreml/cuda/tensorrt/directml) are opt-in cargo features so the default build never risks a from-source cmake. wgpu (one WGSL source -> Metal/Vulkan/DX12/GL/WebGPU) covers fusion/mesh/viewer with no per-platform code. Subgroups/mesh-shaders treated as optional fast-paths behind feature flags, never baseline. Supported: Apple Silicon (Metal/CoreML), Linux+NVIDIA (Vulkan/CUDA/TensorRT), Linux+AMD/Intel (Vulkan), Windows (DX12/DirectML), with a GL fallback for old GPUs. Explicit staging buffers everywhere — never architect around M1 unified memory — so discrete GPUs have no perf cliff. CI matrix: ubuntu (x86_64, libv4l-dev), windows (MSVC), macos-14 (arm64, required since ort 2.0 dropped x86_64 mac prebuilts); default job is pure cargo + clippy + cargo-deny; accel EP builds isolated into opt-in jobs.

**Rust stack:** Inference: ort 2.x (primary, EPs) + candle/candle-transformers (pure-Rust fallback) + tract (CPU floor); ndarray + half for tensor pre/post. GPU-compute: wgpu v29 (+ naga) for unproject/fusion/mesh kernels and viewer. Camera: nokhwa 0.10 (input-native) + fast_image_resize + image. Math: glam (hot path) + nalgebra (SE3/SO3, SVD, Gauss-Newton/LM) + argmin (pose-graph). Mesh: fast-surface-nets + meshopt (FFI, cc-built, no cmake) + transvoxel (seam reference) + ilattice/ndshape (chunk indexing). Concurrency: crossbeam-channel (bounded) + rayon. Viewer: rerun (dev) + three-d + egui/egui-wgpu (shipped). Packaging: hf-hub + directories-next + sha2 (first-run weight download), clap (cli), cargo-deny (license gate), optional pyo3/maturin (offline tooling only). Project license: dual MIT OR Apache-2.0.

**Immature spots:**
- Monocular VO/SLAM has NO production Rust crate (kornia-slam is GSoC-2026 roadmap). The entire depth-assisted direct VO + keyframe management is hand-written on nalgebra/glam primitives — the single biggest engineering item.
- No Rust crate for RANSAC + normal region-growing PLANE detection. The planar differentiator (wall->2 triangles) is custom code (~few hundred lines) on nalgebra SVD.
- No Rust crate for hashed-TSDF/sparse surface fusion or GPU/wgpu isosurface extraction. The sparse-hash fusion and any GPU mesh path are greenfield WGSL/Rust; WGSL exclusive prefix-scan (stream compaction) and cross-backend atomics/subgroup support must be hand-implemented and tested per backend.
- No Rust crate for Hoppe progressive-mesh (vsplit/ecol log). Replaced pragmatically by 're-simplify at monotonically shrinking meshopt target_error' to avoid writing a half-edge engine.
- Loop-closure: DBoW2/3 are C++; need a small Rust HNSW + learned/ORB descriptor (or kornia-bow, alpha) instead.
- ort CoreML-without-cmake is unverified per contradictory docs — must be spiked empirically on the M1 box; CPU/XNNPACK is the guaranteed no-cmake floor if CoreML triggers a from-source build.
- candle's depth_anything_v2 example targets the RELATIVE model; running the METRIC variant in candle means porting the metric head yourself (ort consumes the official ONNX directly).
- Optional DPV-SLAM accel mode requires reimplementing cuda_corr/cuda_ba in wgpu — significant, kept gated.
- Drone path: DJI Mini emits RTMP/UDP H.264 (not UVC), needs a pure-Rust decoder (retina/openh264-rs) behind the camera trait; rolling shutter + autofocus + motion blur stress direct VO and need online intrinsics/robust losses.

**Top risks:**
- Mesh-consistency: planar-first risks visible seams/cracks where plane quads meet progressive patches, and plane mis-merges warp geometry. Mitigations: LockBorder chunking, snap non-planar boundary verts to the plane, robust RANSAC thresholds, transvoxel transition logic if mixed-res seams appear. This is the strategy's headline risk.
- Base-M1 depth latency is INTERPOLATED (no published M1 DAV2 number). If real M1 is >70ms the fps target slips; mitigate with 392/256px input, F16, int8, running depth further below camera fps and leaning harder on multi-frame fusion. Must verify empirically before committing the budget.
- Per-frame depth-model scale inconsistency re-introduces scale drift even though the model is 'metric'. Mitigate with keyframe scale-consistency in the backend + clamped/low-pass global scale; do not trust raw per-frame metric output.
- Unbounded dirty-set on loop closure / drift re-touching old chunks spikes mesh latency. Mitigate with bounded localized re-mesh and amortized cadence.
- ort CoreML/DirectML are NOT MS-prebuilt and CoreML-without-cmake is contradictory in docs — risk of an unexpected from-source cmake build breaking the no-cmake guarantee. Mitigate: CPU/XNNPACK guaranteed floor + empirical spike on M1.
- Hand-rolled VO accuracy below DPV-SLAM; long trajectories without loops still drift. Mitigate: make loop closure first-class, robust Huber losses, optional gated deep mode.
- Drone transfer (rolling shutter, autofocus-changing intrinsics, motion blur, fast 6-DoF) breaks direct photometric VO and indoor-biased depth; needs online intrinsics + outdoor checkpoint + validation — deferred to M4.

**Why this wins:** It is the only candidate that satisfies every hard constraint simultaneously and is the truest realization of the user's vision. Memory: by removing planar inliers from the point pool and storing big surfaces as 2 triangles, peak memory stays well under the 4GB M1 budget (~1.6GB) with no dense TSDF — the leanest option. Algorithmic/CPU-first: the differentiating work (planes, region-grow, progressive simplify) is pure Rust on mature crates (nalgebra, fast-surface-nets, meshopt), minimizing the greenfield WGSL surface that is the biggest Rust-maturity gap, so it ships sooner and more portably. Portability: depth is the only stage with optional accel, all behind one trait with a guaranteed CPU/wgpu floor; fusion/mesh/viewer are identical wgpu/CPU on Apple/NVIDIA/AMD/Windows with no perf cliff and no CUDA/GPL/cmake dependency. Licensing: 100% permissive (dual MIT/Apache) with the only Apache-2.0 metric depth weights, every blocker explicitly excluded and gated. Vision fidelity: 'wall = 2 triangles, complex = many, detail accrues over frames' falls out directly from the plane-registry + observation-driven monotonic refinement design — compute genuinely follows information. The accepted tradeoff (mesh-consistency risk) is concretely mitigated and is the right risk to take for a lean, portable, open-source-clean v1 that still has a gated path to deep-VO accuracy and drone deployment.

### ⚖️ Adversarial verdict: **keep-with-changes**
- realtime feasible: False | fits modest memory: True
- **fps reality:** The 16-22 fps base-M1 number is fiction by the proposal's own admission ("INTERPOLATED... must be verified"). Verified ground truth: DAV2 ViT-S at 518px is 13.3ms on an RTX 4080, and a real user measured ~98-100ms for the V2 small model at 518px across T4/A10/A100/H100 datacenter GPUs. An 8-GPU-core base M1 is a fraction of an RTX 4080. At 392px F16 via CoreML EP the M1 depth stage is realistically 60-110ms, not 45-55ms, so depth-bound throughput is ~9-15 fps at 392px and likely single-digit at 518px. The claimed CoreML EP win is itself unverified (the proposal flags ort-CoreML-without-cmake as contradictory). The pipeline architecture (per-stage threads, bounded triple-buffer channels, object pools, rayon inner loops) is correct and is the right way to hit throughput, but throughput != latency: end-to-end motion-to-photon latency is the sum of stages (~80-140ms), which feeds STALE poses into VO and worsens drift. Realistic honest target: 10-15 fps throughput on base M1 at reduced resolution, ~30 fps on RTX 3060-class with TensorRT. Call it that.
- **memory reality:** The ~1.6GB peak claim is plausible and the planar-first design is genuinely the leanest option (removing plane inliers from the point pool is a real win; no dense TSDF is the right call for 8GB). But 1.6GB is optimistic-accounting: ort with CoreML EP holds the ONNX graph plus CoreML's compiled mlmodelc plus an F16 working set (the 260MB ge-depth line undercounts CoreML's duplicate device-side allocation), the wgpu viewer + three-d + egui easily exceeds the 300MB line once mesh buffers grow in a multi-room session, and the "sparse" fusion hash is only sparse if plane detection actually fires — a cluttered or curved-surface-heavy scene degenerates to the non-planar path and the 400MB fusion line balloons. Realistic peak under load: 2.0-2.8GB. That still fits the ~4GB free-RAM ceiling alongside macOS, so the constraint holds, but the headroom is thin and must be enforced with hard caps on the non-planar cell budget (LRU eviction of out-of-frustum chunks), not assumed.
- **scale/drift:** This is the proposal's central correctness flaw and it is under-mitigated. The claim "scale is owned by the metric model, so no scale ambiguity / no drift" is false. Literature is unambiguous: per-frame learned metric depth is temporally INCONSISTENT (depth range jumps between frames, jitter, and abrupt failures when the scene depth range changes). The model is "metric" only in a population-average sense — DAV2 metric bakes in a training focal length, so on a different webcam intrinsic the output is 10-20% biased AND that bias varies with content. So the design does not remove scale drift; it CONVERTS slow VO scale drift into per-frame scale jitter, then tries to hide it with a clamped, low-pass-filtered global scale factor. That low-pass is a lie that trades responsiveness for stability: it will lag true scale changes (walk into a small room, the global s is still tuned for the hallway) and the median-ratio alignment of VO-triangulated sparse depths against model depths is exactly circular — the VO scale you're trying to anchor is itself derived from the same drifting depth. Without IMU/LiDAR and with a hand-rolled VO (weaker than DROID/DPV-SLAM), a multi-minute handheld session WILL warp, double walls on revisit, and swim, especially on low-texture surfaces (blank walls = the photometric VO's worst case AND where plane detection fires, compounding error). Loop closure helps at loop points but does nothing for open-trajectory exploration (the drone case). BoW loop closure built on a hand-rolled HNSW + ORB descriptors is itself a research-grade effort with no mature Rust crate. The honest framing: this is a best-effort monocular system whose geometry is locally plausible but globally unreliable over minutes; it should be scoped to small bounded spaces with frequent loop closure, and the "no drift" claim must be struck.
- **mesh quality:** The wall=2-triangles vision is elegant and the observation-driven monotonic-shrinking meshopt target_error with hysteresis is a defensible answer to flicker WITHIN the non-planar path. But the headline cracks come from boundaries the proposal hand-waves: (1) plane-quad to surface-nets-patch seams — an oriented bounding rectangle of a plane's extent does NOT share vertices with the marching/surface-nets isosurface of the adjacent non-planar cell, so "snap boundary verts to the plane" requires stitching two fundamentally different meshing topologies, which has no crate and is genuinely hard (T-junctions, normal discontinuities). (2) Plane mis-merge / over-segmentation: RANSAC + normal region-grow on NOISY metric depth will (a) merge two near-coplanar walls into one warped quad, or (b) oscillate plane membership frame-to-frame, making the 2-tri quad jump — visible flicker the hysteresis story doesn't cover because it only governs subdivision, not plane membership. (3) The "subtract expected depth-noise floor from variance before driving subdivision" is the right instinct but DAV2 noise is not a simple floor — it is structured (edge-fattening, sky/low-texture blowups), so flat walls at grazing angles will wrongly stay high-res or planes will eat real geometry. meshopt LockBorder gives crack-free SAME-resolution seams only; the moment plane-quad meets patch you are in mixed-resolution territory and transvoxel is cited only as "reference," i.e. unbuilt. This is buildable but it is the single largest greenfield engineering item after VO, and the proposal correctly self-identifies it as the headline risk.

## HYBRID-PIPELINED Rust core. Six staged threads connected by bounded crossbeam channels (cap 2-3 = backpressure + triple-buffer), with object pools recycled via return channels so peak RAM stays flat under the 8GB M1 ceiling. Stage 1 capture (nokhwa). Stage 2 metric depth via the ort crate behind a DepthBackend trait: ort registers EPs in priority order (CoreML on macOS, CUDA/TensorRT on NVIDIA, DirectML on Windows) with a guaranteed CPU/XNNPACK floor; a pure-Rust candle impl and a tract CPU impl sit behind the same trait as no-C++-toolchain fallbacks. Stage 3 metric-depth-assisted RGB-D-style visual odometry hand-rolled in Rust (nalgebra SE3 + Gauss-Newton direct alignment), with a keyframe pose-graph + loop-closure backend for drift. Stages 4-5 are the novelty: an adaptive spatial-hashed TSDF where per-block voxel resolution is chosen by Welford online TSDF-variance (MrHash idea), fused via wgpu compute, then a 3-kernel GPU marching-cubes extractor (active-detect -> prefix-scan compaction -> vertex-emit, Will-Usher pipeline) run ONLY over dirty blocks, with on-the-fly RANSAC plane merge collapsing low-variance regions to ~2 triangles and meshopt_simplify (LockBorder, ErrorAbsolute) tightening per chunk as observations accumulate. All fusion/mesh GPU work is pure wgpu/WGSL (one codebase, no perf cliff); native accelerators live ONLY behind the depth backend trait. Stage 6 viewer (rerun for dev, three-d+egui for shipped). M0 ships the smallest loop that puts a live triangle mesh on screen and builds via cargo on macOS+Linux.

> Portable Rust monocular-RGB to live adaptive triangle mesh: ort/ONNX metric depth (optional CoreML/CUDA EPs) -> depth-assisted Rust VO -> variance-adaptive spatial-hashed TSDF fused in wgpu -> dirty-block GPU marching cubes + RANSAC plane merge + meshopt, all behind clean backend traits with a portable wgpu/CPU fallback.

**Data flow:** Camera (nokhwa V4L2/AVFoundation/MediaFoundation) -> RGB8 frame (~1-6MB) into a pooled buffer; color-convert YUYV/NV12->RGB via rayon or compute shader → Frame -> resize to 392-518px (fast_image_resize SIMD) -> normalized F16 tensor (ndarray/half) → Tensor -> DepthBackend (ort EP-selected: CoreML/CUDA/DirectML/CPU) -> metric depth map; scale-correction stage applies intrinsics + SLAM-recovered scale (treat DAV2 metric as scale-approximate) → Depth + previous keyframe -> PoseEstimator (Rust direct RGB-D alignment, nalgebra SE3 Gauss-Newton) -> 6-DoF pose; keyframe pose-graph + loop closure corrects drift → Depth + pose -> wgpu unproject compute -> metric point set; integrated into spatial-hashed TSDF blocks intersecting the view frustum near the surface (single command-encoder submission, no intermediate readback) → Per touched block: Welford online TSDF-variance updated -> chooses block voxel resolution (coarse for low variance/walls, fine for high variance/edges); block marked dirty → Dirty blocks -> RANSAC/region-grow plane pass emits confirmed large planes as 2-triangle quads; non-planar dirty blocks -> 3-kernel GPU marching cubes (active-detect -> prefix-scan compaction -> vertex-emit) with Transvoxel transition cells across LOD boundaries → Extracted mesh deltas -> meshopt_simplify per chunk (LockBorder + ErrorAbsolute, target_error shrinks monotonically with observation count) -> double-buffered vertex/index buffers → Mesh deltas -> viewer stage (rerun log out-of-process for dev, or in-process three-d+egui) -> live triangle mesh on screen → Only the camera frame upload and final small mesh-delta readback cross PCIe on discrete GPUs; near-zero on M1 unified memory

**Target fps:** Base M1 (8-core GPU, 8GB unified): ~18-25 fps end-to-end, gated by the pipelined depth stage (~40-55ms DAV2-Metric-Small F16 @392-518px via ort+CoreML; M1 number is INTERPOLATED from M3 Max ~25ms / iPhone ~34ms and MUST be measured before committing). Fusion 3-8ms + mesh-extract 3-8ms run on separate stages so they do not add to the depth-bound frame time; depth can run at 15-20Hz with progressive fusion filling detail. Mid Linux+NVIDIA (e.g. RTX 3060-4070): ~30-60 fps, depth ~13-30ms via CUDA/TensorRT EP (verify empirically — vendor 13ms@RTX4080 is disputed, independents saw up to ~98ms on some GPUs), fusion/mesh sub-5ms in wgpu/Vulkan, VO ~5ms; here the pipeline can be camera-frame-rate-bound rather than depth-bound. Honest caveat: SOTA learned dense SLAM systems are NOT real-time on base M1, which is exactly why this design uses a small quantized depth net + frame-to-model tracking + dirty-block extraction instead of global dense BA.  
**Peak memory:** 2310 MB  
**Platforms:** macOS 13.4+ Apple Silicon (M1 baseline) — wgpu/Metal core; ort+CoreML(ANE) optional, CPU/candle/tract fallback, Linux x86_64 with NVIDIA — wgpu/Vulkan core; ort+CUDA/TensorRT optional, XNNPACK/CPU fallback, Linux x86_64 with AMD/Intel GPU — wgpu/Vulkan core; CPU/candle fallback (no vendor accel needed for fusion/mesh), Windows 10/11 x86_64 — wgpu/DX12 core; ort+DirectML optional, CPU fallback, Intel Mac (best-effort) — wgpu/Metal core but ort 2.0 dropped x86_64 mac prebuilts; use tract/candle pure-Rust depth fallback, WebGPU/Wasm (viewer + portable compute, future) — wgpu WebGPU backend, candle Wasm inference

**Metric scale:** Metric scale is owned UPSTREAM by the learned depth model, not the VO — this is the key move that sidesteps classic monocular scale ambiguity. Ship DAV2-Metric-Small (Apache-2.0, ViT-S, 24.8M, F16 ONNX): indoor (Hypersim/20m) and outdoor (VKITTI/80m) checkpoints, selected per deployment (drone -> outdoor). BUT the brief is explicit that DAV2 metric output is SCALE-APPROXIMATE: it bakes in a ~470mm-class training focal length and drifts 10-20% when real intrinsics differ, and indoor/outdoor checkpoints are not interchangeable. So a mandatory scale-correction stage sits between depth and fusion: (1) when camera intrinsics are known/calibrated, rescale depth by the focal ratio; (2) recover a global scale factor from SLAM triangulation/keyframe geometry and apply per-keyframe; (3) enforce inter-keyframe scale consistency in the pose-graph backend so per-frame depth-model jitter does not re-introduce scale drift. As a permissive alternative worth evaluating in offline tooling, Metric3D-V2-small is BSD-2 and predicts metric directly from intrinsics, but BSD 'non-commercial' wording is ambiguous — keep DAV2-Small Apache-2.0 as the shippable default. For the unknown-intrinsics drone, evaluate UniDepth-style intrinsics prediction in offline Python EVAL only (CC-BY-NC, cannot ship).

**Pose:** No mature Rust SLAM exists (kornia-slam is GSoC-2026 roadmap), so VO is hand-rolled — and the metric depth makes this tractable. PoseEstimator trait with three impls: (a) DEFAULT portable native-Rust depth-assisted direct VO — turn the monocular problem into RGB-D using the metric depth, estimate frame-to-keyframe 6-DoF by minimizing photometric + inverse-depth/geometric residual via nalgebra SE3 Lie-algebra Gauss-Newton/LM (DSO-style math, reimplemented, NOT GPL-imported); (b) optional wgpu-accelerated patch VO; (c) optional CUDA DPVO/DPV-SLAM FFI (MIT, license-clean) for NVIDIA boxes only — its cuda_corr/cuda_ba kernels would need wgpu reimplementation to be portable, so it stays a gated NVIDIA-only fast path. Drift is handled as a first-class concern (no IMU = loop closure is the only long-trajectory anchor): a keyframe pose-graph backend (argmin/nalgebra LM, optionally cxx-FFI to BSD GTSAM) with loop detection via a small Rust HNSW over a learned global descriptor (avoid GPLish DBoW2). Enforce keyframe scale consistency to stop depth-model drift leaking back as scale drift. AVOID ORB-SLAM3 (GPLv3 viral), DROID-SLAM (>=11GB VRAM, CUDA-only), MASt3R-SLAM (CC-BY-NC weights) — design references only.

**Fusion & meshing:** Substrate is spatial voxel-hashing (Niessner) + TSDF: hash table of 8^3 blocks allocated only in occupied near-surface space, flat storage buffers + integer hashing = WGSL-native, no octree pointer-chasing. Adaptive LOD is driven from the TSDF/voxel domain (cleaner and more incremental than mesh decimation): per-block Welford online TSDF variance (MrHash, ToG 2025) picks voxel resolution — low variance (a wall) -> coarse block -> few MC cells; high variance (an edge) -> fine block. Critically, subtract an expected depth-noise floor before using variance as the criterion, because monocular learned-depth noise inflates variance and would wrongly keep walls high-res. Incremental update: integrate a depth frame only into frustum-near blocks (a few hundred us to ~2ms), mark touched blocks dirty. Extraction NEVER runs over the full volume (a dense 256^3 MC is ~250ms on M1) — only dirty blocks, on a separate cadence, via the ported Will-Usher 3-kernel WGSL pipeline: active-voxel detect -> exclusive prefix-scan stream compaction (the one non-trivial WGSL primitive, hand-written) -> vertex emit with computed offsets, double-buffered to the viewer. Two-tier mesher realizing 'wall->2 tris, complex->many tris': (1) custom RANSAC + normal region-grow (nalgebra SVD total-least-squares) emits confirmed large planar regions as 2-triangle quads; (2) everything else -> GPU MC -> meshopt_simplify per chunk. Crack-free LOD via Transvoxel transition cells across mixed-resolution blocks and meshopt_SimplifyLockBorder for same-resolution chunk independence; protect plane-intersection corners with SimplifyVertex_Priority. Progressive/observation-driven refinement WITHOUT a Hoppe vsplit engine: per-region observation count gates a MONOTONICALLY shrinking meshopt target_error (use SimplifyErrorAbsolute in world units, with hysteresis — never coarsen a stabilized region) so detail accrues over frames and triangles do not pop.

**Performance plan:** Six std::thread stages joined by BOUNDED crossbeam-channels (cap 2-3): capture -> depth -> VO/pose -> fusion -> mesh-extract -> viewer. Bounded channels give automatic backpressure AND act as the triple-buffer (producer fills slot N+1 while consumer reads N). Object-pool pattern: preallocate frame buffers, F16 depth tensors, and vertex/index pools once and recycle via a return channel so no stage ever stalls on the allocator and peak RAM stays flat under the 8GB ceiling. rayon for embarrassingly-parallel inner loops (color-convert, per-chunk meshopt, RANSAC sampling). Depth dominates and is PIPELINED, so end-to-end fps tracks the depth stage as throughput, not added latency; run depth BELOW camera framerate (15-20Hz) and let progressive multi-frame fusion integrate detail. GPU discipline: chain unproject->fuse->MC into a single command-encoder submission; only the ~1-6MB frame upload and the small final mesh-delta readback cross PCIe (sub-ms at PCIe3 x16); never read back intermediate depth/TSDF; map_async always followed by device.poll() on native. Budget a conservative 1.2-1.6x wgpu-vs-native multiplier and skip WGSL validation in release (create_shader_module_unchecked). Profile every backend with wgpu timestamp queries. Optional-accel split: CoreML/CUDA/DirectML live ONLY behind the depth backend trait; fusion+mesh are pure wgpu everywhere so there is no accelerator perf cliff off Apple Silicon. Per-stage M1 budget (~20-25fps pipelined): capture+convert 3-4ms; depth 40-55ms (throughput-pipelined); upload <1ms; VO 5-10ms; fusion 3-8ms; mesh-extract 3-8ms; viewer overlapped.

**Portability plan:** Two backend traits give optional-accel-with-fallback and no perf cliff. (1) DepthBackend: default ort impl selects EP at runtime — CoreML on macOS, CUDA/TensorRT on NVIDIA, DirectML on Windows — with an always-present XNNPACK/CPU floor via ort's per-operator EP fallback chain; accelerators are cargo features inside ge-depth so a plain `cargo build` compiles the portable CPU path on every OS. A second candle impl (pure-Rust, Metal/CUDA/CPU, no C++ ONNX Runtime) and a tract impl (pure-Rust CPU, zero native toolchain) sit behind the same trait for the no-cmake/no-binary path. (2) ComputeBackend/PoseEstimator: fusion and mesh extraction are pure wgpu/WGSL (Metal/Vulkan/DX12/GL/WebGPU from ONE source via naga), so the GPU geometry pipeline is identical on Apple Silicon, Linux+NVIDIA, Windows, and AMD — accelerators are deliberately NOT used here. Do NOT architect around unified memory: M1 enables zero-copy (nokhwa output-wgpu) but discrete GPUs pay staging/PCIe, so explicit staging buffers + out-of-core block streaming are built from day one. CaptureSource trait wraps nokhwa so a drone H.264 decoder (retina/openh264-rs) or file source swaps in without touching ge-core. CI: GitHub Actions matrix ubuntu (apt libv4l-dev) + windows (MSVC) + macos-14 arm64 (ort 2.0 dropped x86_64 mac prebuilts), default job pure cargo (clippy, test, cargo-deny), optional accel EPs isolated to opt-in jobs where C/cmake may intrude. Dual MIT OR Apache-2.0 license, cargo-deny blocks GPL/CC-BY-NC creep. **Top risks:**
- Base-M1 depth latency is INTERPOLATED (no published M1 DAV2 number) and vendor/CoreML benchmarks are unreliable (CoreML EP does not accelerate every op on ANE; some fall back to GPU/CPU). If real M1 depth is >60ms the 18-25fps target slips — must measure on the actual box in M0/M2 before committing the budget.
- ort CoreML/DirectML have no Microsoft prebuilt binaries; enabling the CoreML feature may trigger a slow from-source cmake build, breaking the no-cmake guarantee. Spike-verify on the M1 box; keep tract/candle as the zero-toolchain floor.
- The three biggest pieces (VO/SLAM, hashed-TSDF + GPU marching cubes incl. prefix-scan, RANSAC plane + progressive meshing) have NO mature Rust crate — this is mostly greenfield engineering, the dominant schedule risk. WGSL prefix-scan + atomics/subgroup support varies across Metal/Vulkan/DX12/GL (portability cliff if untested on GL).
- Dirty-set blowup: drift or loop closure re-touching old blocks can grow the dirty set unbounded and spike extraction latency. Bound the per-frame extraction budget and cap re-extraction; never extract the full volume.
- Monocular depth noise inflates TSDF variance, so the adaptive-LOD criterion can wrongly keep flat walls high-res; needs an empirically-tuned depth-noise floor. Per-frame depth-scale inconsistency can re-introduce scale drift unless keyframe scale consistency is enforced in the backend.
- 8GB unified ceiling (macOS takes 3-4GB): peak ~2.3GB is comfortable ONLY if all per-frame buffers are pooled/recycled and TSDF blocks stream out-of-core; per-frame heap churn would fragment and stall. Discrete-GPU portability requires the staging/streaming path from day one — must not architect around unified memory.
- Drone transfer (DJI Mini): rolling shutter, autofocus (changing intrinsics), motion blur, fast 6-DoF motion break direct/photometric VO and the indoor-biased depth model; H.264/RTMP is not UVC so needs a separate decoder. Treat as a later milestone with online-intrinsics + robust losses, validated separately.

**Why this wins:** It is the only design that satisfies ALL of the project's hard constraints simultaneously and idiomatically. Portability-first with no perf cliff: fusion and mesh extraction are pure wgpu/WGSL (one codebase -> Metal/Vulkan/DX12/GL/WebGPU), and native accelerators (CoreML/CUDA/DirectML) are confined behind a single DepthBackend trait with a guaranteed CPU/pure-Rust (candle/tract) floor, so a plain `cargo build` runs everywhere and accelerated boxes just go faster. It dissolves the hardest monocular problem — metric scale — by owning scale UPSTREAM in a permissively-licensed (Apache-2.0 DAV2-Small) depth model and feeding RGB-D-style VO, sidestepping both monocular scale ambiguity AND the GPLv3/CUDA/non-commercial traps that disqualify ORB-SLAM3, DROID/DPVO, and MASt3R. The adaptive variance-driven TSDF + dirty-block GPU marching cubes + RANSAC plane merge directly realizes the 'compute follows information' thesis (wall->2 tris, edges->many, detail progressive) while NEVER extracting the full volume — the single thing that keeps a base-M1 8-core GPU and 8GB ceiling viable. The threaded bounded-channel + object-pool pipeline keeps peak memory flat and turns the dominant depth latency into throughput, not added latency. Every dependency and every shippable weight is MIT/Apache/BSD permissive with cargo-deny enforcement, so it is genuinely open-source and runs for anyone — Linux+NVIDIA, Windows, AMD, and Apple Silicon — not Mac-only.

### ⚖️ Adversarial verdict: **keep-with-changes**
- realtime feasible: False | fits modest memory: True
- **fps reality:** The 18-25fps end-to-end claim is unproven and the framing is misleading. The single load-bearing number — DAV2-Metric-Small at 40-55ms on a base 8-GPU-core M1 — is INTERPOLATED from an M3 Max (~25ms) and an iPhone NPU (~34ms); no published base-M1 ONNX figure exists (confirmed: searches return only NVIDIA/Jetson/MLX numbers). The base M1 GPU is ~2-2.6 TFLOPS vs the M3 Max's ~14; a 2-3x slowdown to 60-110ms is at least as likely as 50ms, which alone caps the pipeline at 9-16fps. The CoreML EP does NOT run every ViT op on the ANE — attention/layernorm patterns fall back to GPU/CPU and partition-boundary copies add unbudgeted latency. The pipelined-throughput argument is technically true but hides serialized costs: VO is photometric Gauss-Newton over a full depth map (8-10ms is optimistic for multi-level robust LM with outlier reweighting — 15-30ms realistic), and the WGSL prefix-scan + MC + meshopt chain contends with depth inference for the same 8-core GPU and memory bandwidth. Honest expectation on base M1: 8-15fps depth-bound with a usably-progressive mesh, NOT 18-25. Mid-NVIDIA 30-60fps is plausible. Mark the fps headline UNKNOWN-until-measured.
- **memory reality:** 2.31GB peak is plausible-to-tight only if the discipline is real, and it has hidden line items. macOS 26.x on 8GB realistically leaves ~3.5-4.5GB after WindowServer/system; 2.31GB fits with ~1GB headroom — acceptable, not comfortable. Underweighted risks: (a) ort+CoreML keeps BOTH an ONNX Runtime arena AND a CoreML/MPSGraph compiled graph resident; 420MB for ge-depth is optimistic — budget 500-700MB. (b) Two wgpu devices (core + three-d/rerun) on unified memory double-count GPU runtime; share one device. (c) rerun out-of-process is honest but its 300-800MB is NOT in the 2.31GB, so dev mode can blow the 8GB box even if shipped fits. (d) 700MB TSDF assumes aggressive out-of-core streaming that is itself unbuilt greenfield; a naive resident hash table over a multi-room walk will exceed it. Verdict: fits for the shipped three-d path with strict pooling; the rerun dev path needs 16GB and should be documented as such.
- **scale/drift:** The correctness crux, and the mitigations are weaker than the confident tone implies. Owning scale upstream in the depth model is the right move and sidesteps classic monocular scale ambiguity per-frame — but it converts a global-gauge problem into a per-frame metric-CONSISTENCY problem, arguably worse for a clean mesh. DAV2-metric output is per-frame affine-inconsistent: the same wall across 200 frames breathes 5-20% with exposure, focus, angle, and texture. Fusing that into a TSDF produces exactly the wall-warp, surface-thickening, and ghost-doubling the brief fears, and pose-graph scale-consistency cannot fix it because the error is non-rigid, not a single scalar. Direct/photometric RGB-D VO on PREDICTED (not sensed) depth compounds it: tracking and map share the same depth error and reinforce it. With NO IMU and loop closure deferred to M4, a multi-minute free walk WILL drift in pose and the mesh WILL swim and double on revisit; loop closure fixes revisit seams but leaves a rubber-sheeted interior. Required: (1) a per-keyframe affine (scale+shift) alignment against the existing TSDF BEFORE fusion — the single most important missing stage; (2) depth-confidence-weighted integration, not uniform; (3) ship loop closure in M1/M2, not M4; (4) set expectations honestly — locally-metric, room-scale-coherent, globally-drifting, NOT survey-accurate.
- **mesh quality:** The two-tier planar+MC mesher is sound in principle but bleeds at the seams, which the proposal hand-waves. (1) Transvoxel handles mixed-RESOLUTION block seams, but the boundary between a RANSAC plane-quad and an adjacent MC patch is a DIFFERENT seam Transvoxel does not solve — a 2-tri quad meeting a dense MC patch T-junction-cracks and z-fights unless explicitly stitched (unbudgeted). (2) RANSAC plane fitting on noisy monocular depth will FLICKER between planar (2 tris) and non-planar (200 tris) frame-to-frame as depth jitters across the inlier threshold; monotonic-shrinking-error + hysteresis stabilizes settled regions but not the initial oscillation while a wall is still being observed — visible popping. (3) never-coarsen-stabilized-region directly conflicts with loop-closure pose correction: when closure shifts geometry, stabilized regions MUST re-mesh or you get doubled surfaces — that override logic is unspecified. (4) meshopt LockBorder gives chunk independence but locked-border vertices accumulate over a long session, degrading simplification. The wall-to-2-triangles demo looks great static and reveals all four problems the moment the camera moves continuously. Achievable, but it is M3 + M4 + a seam-stitching milestone, not one M3.
- **portability:** Genuinely the best part — but not as clean as claimed. wgpu-everywhere for fusion/mesh is correct and the right way to avoid a CUDA-lock perf cliff; confining accelerators behind DepthBackend is idiomatic. Cracks: (1) GL in the matrix is a liability — wgpu's GL/WebGL backend has no/limited compute, so the 3-kernel MC + prefix-scan CANNOT run there; one-WGSL-source-everywhere is FALSE for the compute path. Drop GL from the compute story; use the fast-surface-nets CPU mesher as that fallback. (2) Subgroup-accelerated prefix-scan: confirmed wgpu subgroups are full only on Vulkan, conditional on Metal/DX12, absent on GL/old drivers — so the prefix-scan needs a non-subgroup workgroup-scan fallback, so it is NOT literally one source. (3) ort 2.x dropped x86_64-mac prebuilts, pushing Intel Macs onto tract/candle, and candle's Metal backend has confirmed missing-op gaps (e.g. BF16 matmul) — the Intel-Mac path is real but slow and partly unproven. Licensing: dependency discipline (dual MIT/Apache + cargo-deny) is exemplary. BUT the flagship weight claim is OVERSTATED: DAV2-Small weights are Apache-2.0 yet upstream issue DepthAnything/Depth-Anything-V2 #320 flags UNRESOLVED data-derived license risk (training-data provenance not cleared for commercial use, no maintainer response). Metric3D-V2 BSD-but-contact-for-commercial is correctly flagged. Net: portable architecture YES; ships-clean-weights-for-anyone-including-commercial NOT YET.
- **Rust maturity:** Crate selection is the most credible part, but the proposal's own immatureSpots list IS the story and it is large. PRODUCTION-READY: wgpu 29.x (mature for graphics+compute), nalgebra/glam, crossbeam/rayon. nokhwa works but is quirky (uneven V4L2/AVFoundation/MediaFoundation; macOS format negotiation and continuity-camera are known pain). ort 2.x is a production-grade runtime but rc/not-API-stable (pin exact); official arm64-mac prebuilts with CoreML now exist which de-risks the no-cmake story, but it MUST be spiked on the actual M1 in M0, not assumed (cmake confirmed absent on the dev box, clang present at /usr/bin/clang). APIRATIONAL/GREENFIELD where Rust bites hardest: (a) SLAM — no production Rust monocular VO/SLAM; kornia-slam is a 2026 roadmap, Rust-CV is semi-maintained. The hand-rolled direct VO + pose-graph + loop closure is DSO-class numerical work (robust kernels, linearization, keyframe mgmt, gauge handling) budgeted at 15-25 days — off by 2-4x. (b) hashed-TSDF + GPU MC + hand-written exclusive prefix-scan is greenfield with a real cross-backend (atomics/subgroup) test-matrix burden. (c) RANSAC plane + progressive re-simplify is custom. (d) candle as fallback is weaker than presented: the candle example targets the RELATIVE model, so the metric head must be ported, AND candle Metal has confirmed op gaps — the pure-Rust fallback is itself a project. meshopt via cc (no cmake) is fine.
