# Gods Eye — Architecture (Recommended)
_Final synthesis of the design workflow. This is the plan of record._

## Recommended architecture

Six-stage pipelined Rust core, one std::thread per stage joined by bounded crossbeam channels (cap 2-3, doubling as backpressure + triple-buffer), with object-pooled buffers recycled via return channels so peak RAM stays flat. Stages: (1) capture (nokhwa, with a file/video source as the primary dev input); (2) metric depth via ort/ONNX (DAv2-Metric-Small ViT-S) behind a DepthBackend trait with runtime EP selection; (3) depth-assisted RGB-D-style direct visual odometry hand-rolled on nalgebra SE3 + Gauss-Newton, with a per-keyframe affine (scale+shift) depth-alignment stage BEFORE fusion, a keyframe pose-graph, and loop closure brought forward as first-class; (4) sparse spatial-hashed TSDF fused in wgpu compute with depth-confidence-weighted integration; (5) dirty-block mesh extraction — fixed-resolution first (GPU marching cubes via a hand-written workgroup-scan, with a fast-surface-nets CPU equivalent as the true portable floor), then layered planar-RANSAC 2-triangle quads + per-chunk meshopt_simplify(LockBorder, ErrorAbsolute) as a post-process; (6) viewer (rerun out-of-process for dev, three-d+egui for shipped). All fusion/mesh GPU work is pure wgpu/WGSL (one codebase, no per-platform accel branch); native accelerators (CoreML/CUDA/TensorRT/DirectML) live ONLY behind the DepthBackend trait, always with a CPU/wgpu floor. This is essentially proposal #1 (TSDF-baseline-then-simplify) hardened with the cross-cutting adversarial required-changes (measure-before-budget, affine depth alignment, confidence-weighted fusion, early loop closure, plane/voxel seam-stitching as its own task, GL dropped from the compute matrix, honest sub-15fps framing).

## Rationale

All three proposals converged on the identical spine (nokhwa -> ort depth -> hand-rolled depth-assisted VO -> sparse-hash TSDF in wgpu -> dirty-block extraction + adaptive post-process -> viewer) and the identical adversarial verdict (realtime=false but live-at-depth-cadence, fitsMem=true, scale/drift is the central research risk). I pick proposal #1's sequencing because it is the most honest about ordering: get a CORRECT fixed-resolution live mesh on screen across two platforms FIRST (M0), then layer adaptivity as a deliberate post-process. Proposal #2 (planar-first) is the leanest on memory and truest to wall=2-triangles, but it front-loads the unsolved plane-quad-to-patch seam-stitching and a degenerate-to-non-planar memory blowup; proposal #3 front-loads variance-adaptive (MrHash) LOD whose depth-noise-floor tuning is unvalidated on monocular depth. Both #2 and #3 put their hardest, least-crate-supported novelty on the critical path. #1 isolates risk: the wgpu compute path is pure and uniform (its strongest portability decision), and the genuinely greenfield, research-grade work (VO + scale/drift) is sequenced as a dedicated milestone with an offline eval harness rather than wired blind into the live loop. I fold every reviewer required-change into #1: stop quoting 18-25fps; add the per-keyframe affine depth-alignment stage (the single most important missing piece for mesh consistency); confidence-weighted TSDF integration; loop closure promoted to M2; plane-quad/voxel seam-stitch as an explicit gated task; GL dropped from the compute matrix with fast-surface-nets CPU as that fallback; DAv2-Small reclassified as provenance-unresolved.

## Key picks

### Depth model
Depth Anything V2 Metric-Small (ViT-S, 24.8M params) — Apache-2.0 on BOTH code and the Small weights (the only DAv2 tier that is permissive; all Base/Large/Giant and metric tiers above Small are CC-BY-NC-4.0). Ship two domain checkpoints: Indoor (Hypersim/20m) and Outdoor (VKITTI/80m), selected per deployment (drone=Outdoor). Exported to ONNX (opset 17/18, F16). Rust inference runtime: ort 2.x (ONNX Runtime bindings) as PRIMARY, with candle (pure-Rust, Metal/CUDA/CPU) and tract (pure-Rust CPU) as fallbacks behind the same trait. CRITICAL LICENSE CAVEAT TO DOCUMENT IN NOTICE, NOT ASSERT AWAY: upstream issue DepthAnything/Depth-Anything-V2 #320 flags unresolved training-data provenance (Hypersim/VKITTI EULA/dataset licenses) for commercial use — treat as 'permissive label, unverified data provenance,' keep --model-path for a cleared-weight swap, and identify a clean-provenance alternative. Treat output as SCALE-APPROXIMATE (10-20% focal-biased) and per-frame affine-inconsistent, never as ground-truth metric.

### Pose / VO
Native-Rust depth-assisted RGB-D-style direct visual odometry behind a PoseEstimator trait. The learned metric depth turns monocular into RGB-D, so pose is frame-to-keyframe SE3 estimated by minimizing a robust (Huber) photometric + inverse-depth/geometric residual via nalgebra SE3 Lie-algebra Gauss-Newton/LM on a coarse-to-fine pyramid; glam on the hot inner loop, rayon for residual/Jacobian accumulation; sparse FAST/ORB + PnP-RANSAC for re-init on tracking loss. CRITICAL HARDENING (from adversarial review): a per-keyframe affine (scale+shift) alignment of predicted depth against the existing TSDF runs BEFORE fusion (DAv2 depth is affine-inconsistent frame-to-frame, not metric-absolute), and depth-scale is a per-keyframe latent jointly handled in the pose-graph rather than a single global low-pass scalar. Drift: keyframe pose-graph backend (argmin/nalgebra LM, optional cxx-FFI to BSD GTSAM) with loop closure promoted to M2 as first-class (small Rust HNSW over a learned global descriptor; avoid C++ DBoW2). Optional gated NVIDIA-only accel: cxx-FFI to MIT DPVO/DPV-SLAM, or a wgpu patch-VO reimpl — never required. AVOID as deps: ORB-SLAM3/DSO (GPLv3 viral even via FFI), DROID/MASt3R-SLAM (CUDA + non-commercial weights) — design references only. Honest scope: locally-metric, room-scale-coherent, globally-drifting; reliable in small bounded spaces with frequent loop closure.

### Fusion & meshing
Sparse voxel-hashing + TSDF substrate (Niessner: flat hash table of 8^3 blocks allocated only in occupied near-surface space, integer hashing = WGSL-native, no octree pointer-chasing), with out-of-core block streaming + LRU eviction of out-of-frustum chunks built from day one (mandatory for both 8GB unified and discrete VRAM). Depth-confidence-weighted integration (down-weight edges/far-field/low-texture) so monocular noise does not corrupt flat surfaces. Extraction NEVER over the full volume (dense 256^3 MC ~250ms on base M1) — only dirty blocks, on a separate cadence. SEQUENCING (the key decision): ship FIXED-resolution extraction first (GPU marching cubes ported from the Will-Usher 3-kernel pipeline with a hand-written workgroup-scan prefix-scan, plus a fast-surface-nets CPU mesher as the portable floor where wgpu compute is weak), double-buffered to the viewer. Then layer adaptivity as a deliberate post-process: (1) custom RANSAC + normal region-grow (nalgebra SVD) emits confirmed large planar regions as 2-triangle quads; (2) per-chunk meshopt_simplify with SimplifyLockBorder (crack-free same-res) + SimplifyErrorAbsolute (world-unit, scale-stable) + SimplifyVertex_Priority (protect plane corners); (3) observation-driven progressive refinement via a MONOTONICALLY shrinking target_error with hysteresis (never coarsen a stabilized region) — EXCEPT a topology-change override that forces re-mesh when loop closure shifts geometry. Variance-adaptive (MrHash/Welford) LOD is demoted to a feature-flagged research spike, not the critical path. EXPLICIT GATED TASKS before mixed adaptivity ships: a plane-quad-to-voxel-patch seam-stitch pass (Transvoxel does NOT solve that boundary), plane-MEMBERSHIP hysteresis (not just subdivision hysteresis) to stop quad flicker, and WGSL Transvoxel transition cells validated in isolation for mixed-resolution seams.

### Rust stack
Inference: ort 2.x (PRIMARY, EP cargo features coreml/cuda/tensorrt/directml/xnnpack-cpu, download-binary to avoid cmake) + candle/candle-transformers (pure-Rust fallback) + tract (pure-Rust CPU floor); ndarray + half for F16 pre/post. GPU compute: wgpu 29.x + naga (WGSL: unproject, TSDF integrate, dirty-block marching cubes with hand-written workgroup-scan prefix-scan). Camera: nokhwa 0.10 (input-native) + image + fast_image_resize; retina/openh264-rs later for drone H.264 (separate milestone). Math/pose: nalgebra (SE3/SO3, SVD, Gauss-Newton/LM), glam (SIMD hot path), argmin (pose-graph solver). Mesh: meshopt (cc-built FFI, QEM simplify + LockBorder + ErrorAbsolute + VertexPriority), fast-surface-nets (CPU isosurface fallback + portable floor), transvoxel (LOD-seam reference), ilattice + ndshape (chunk indexing), block-mesh (greedy coarse-quad option). Threading: std::thread + crossbeam-channel (bounded, backpressure + triple-buffer) + rayon (data-parallel inner loops). Viewer: rerun/re_sdk (dev, out-of-process gRPC) + three-d + egui/egui-wgpu (shipped). Packaging: hf-hub + directories-next + sha2 (first-run weight download), clap (CLI), cargo-deny (license gate), pyo3+maturin (optional offline tooling only). Optional FFI: cxx (only for BSD GTSAM pose-graph or MIT DPV-SLAM CUDA — never GPLv3). Every core crate is MIT/Apache/BSD permissive.

### Backend abstraction
Two clean Rust traits give optional-accel-with-fallback and no perf cliff. (1) `trait DepthBackend { fn infer(&self, frame: &DepthInput) -> Result<DepthMap>; }` with impls: OrtBackend (default; ort registers EPs in priority order [CoreML on macOS, CUDA/TensorRT on NVIDIA, DirectML on Windows] with an always-present XNNPACK/CPU floor via ort's per-operator EP fallback chain — the EP chain literally IS the trait fallback), CandleBackend (pure-Rust Metal/CUDA/CPU, no C++ runtime), TractBackend (pure-Rust CPU, zero native toolchain). Accelerators are cargo features inside the ge-depth crate, so a plain `cargo build` compiles only the portable CPU path on every OS — no combinatorial release churn, no accidental cmake. (2) `trait PoseEstimator` and `trait ComputeBackend`: the native-Rust direct VO and the pure-wgpu fusion/mesh are the ALWAYS-AVAILABLE defaults; optional wgpu patch-VO and optional CUDA DPV-SLAM FFI sit behind PoseEstimator for accel boxes but are never required. Critically, fusion/mesh/viewer GPU work uses NO optional-accel branch — it is one WGSL source (naga -> MSL/SPIR-V/HLSL) so there is zero per-platform perf cliff there; accelerators are confined strictly to the depth stage. All compute kernels must have a CPU+rayon equivalent (fast-surface-nets etc.) so zero correctness depends on a compute shader, keeping the GL/Intel-Mac/old-GPU path alive. A CaptureSource trait wraps nokhwa so a drone H.264 decoder or file source swaps in without touching ge-core.

**Target fps (do not over-commit):** UNKNOWN until M0 empirical measurement — do NOT quote 18-25fps. Honest design target on base M1: 10-15 fps depth-bound (live-at-depth-cadence), with the mesh updating smoothly via progressive multi-frame fusion even though depth runs below camera rate (~12-18Hz). Verified ground truth that caps this: DAv2-Small is ~13ms@518px on an RTX-4080 but a real user measured ~98-100ms on datacenter GPUs; the base M1 8-GPU-core box (~2-2.6 TFLOPS vs M3 Max ~14) realistically lands depth at 60-110ms, so single-digit-to-15fps is the planning number until the M0 spike measures it. Mid Linux+NVIDIA (RTX 3060-4070, CUDA/TensorRT EP): depth ~13-30ms -> pipeline becomes VO/fusion-bound at ~30-60fps. CPU-only floor (tract/candle): ~3-8fps, correctness/CI only.

**Supported platforms:**
- macOS 13.4+ Apple Silicon (M1/M2/M3, arm64) — wgpu/Metal core; ort+CoreML/ANE optional with CPU/XNNPACK fallback; primary dev/baseline box (macos-14 CI)
- Linux x86_64 + NVIDIA — wgpu/Vulkan core; ort+CUDA/TensorRT optional with CPU fallback; needs libv4l-dev for nokhwa V4L2; second reference target (RTX-class)
- Linux x86_64 + AMD/Intel GPU — wgpu/Vulkan core; depth via CPU/XNNPACK (no vendor EP)
- Windows 10/11 x86_64 — wgpu/DX12 core; ort+DirectML optional with CPU fallback; MSVC toolchain
- Intel macOS (best-effort, documented degraded) — wgpu/Metal core but ort 2.0 dropped x86_64 mac prebuilts, so CPU-only tract/candle depth
- Portable floor everywhere — wgpu (Metal/Vulkan/DX12) compute + fast-surface-nets CPU mesher + pure-Rust tract/candle depth, runs with zero native ML toolchain. NOTE: GL/WebGL DROPPED from the COMPUTE matrix (wgpu GL backend lacks usable compute) — GL targets fall back to the CPU mesher path
- WebGPU/Wasm — viewer (rerun/egui) in-browser is a stretch goal; full pipeline browser support is not v1

## Where Rust is immature (and the plan)
- Monocular VO/SLAM has NO production Rust crate (kornia-slam is GSoC-2026 roadmap; Rust-CV arrsac/essential-matrix/PnP are semi-maintained and uneven). PLAN: the entire ge-slam stage (direct alignment, Lie-algebra GN/LM, keyframe pose-graph, loop closure) is hand-written on nalgebra/glam/argmin — NO FFI needed because depth supplies scale. This is DSO-class numerical work; budget it as the dominant line item with its own offline accuracy harness (compare against COLMAP ground truth on recorded sequences) BEFORE wiring live. Keep a cxx-FFI escape hatch to permissive BSD GTSAM for the pose-graph if argmin proves inadequate (cmake-free constraint may force this trade — GTSAM build needs evaluation).
- GPU TSDF fusion AND GPU/wgpu marching cubes have NO mature crate. PLAN: all WGSL kernels hand-authored (port Will-Usher pipeline). The exclusive prefix-scan stream-compaction is the one non-trivial primitive — implement a non-subgroup workgroup-scan fallback because wgpu subgroups are full only on Vulkan, conditional on Metal/DX12, absent on GL. Mine threecrate for the wgpu-TSDF shader approach but do NOT depend on it (low-traction, pre-1.0, no plane detection).
- RANSAC/region-grow PLANE detection and Hoppe progressive-mesh (vsplit/ecol) have NO mature Rust crate. PLAN: RANSAC is ~few hundred lines on nalgebra SVD; progressive refinement done pragmatically via monotonic re-simplify at shrinking meshopt target_error (avoids a half-edge engine). Add plane-MEMBERSHIP hysteresis and minimum-support gating to stop quad flicker.
- Plane-quad-to-voxel-patch seam-stitching has NO crate and Transvoxel does NOT solve that boundary (different topologies, T-junctions). PLAN: explicit gated milestone task — snap patch border vertices onto the plane basis, prove crack-free on a wall-meets-clutter scene; until then ship QEM-LockBorder-only adaptivity.
- ort CoreML-without-cmake is UNVERIFIED (contradictory sources; MS ships no prebuilt CoreML/DirectML). PLAN: spike-verify on the actual M1 box in week 1 (cmake confirmed absent, clang at /usr/bin/clang present). If it triggers a from-source cmake build, fall back to tract/candle immediately and gate no milestone on CoreML. CPU/XNNPACK is the guaranteed no-cmake floor.
- candle's depth_anything_v2 example targets the RELATIVE model + candle Metal has confirmed op gaps — the pure-Rust metric fallback means porting the metric head yourself. PLAN: ort consumes the official ONNX directly with less work; treat candle metric as a funded sub-project or rely on ort+tract.
- nokhwa is single-maintainer with documented AVFoundation/NV12->RGB quirks on M1 (the baseline box). PLAN: make a file/video source the primary dev input, pin a known-good nokhwa version, assert frame format/byte-size at the capture boundary so capture bugs surface immediately rather than as phantom pipeline failures.

## What this CAN do (inspired-by-Tesla, honestly)
- ✅ Produce a live, incrementally-built triangle mesh of the environment from a single RGB camera with NO LiDAR/depth-sensor/IMU — pure vision geometry
- ✅ Recover locally-metric, room-scale-coherent geometry: walls, floors, large planar surfaces collapsed to ~2 triangles; complex regions kept dense; detail added progressively over multiple frames (compute follows information)
- ✅ Run genuinely cross-platform from one codebase (Apple Silicon/Metal, Linux+NVIDIA/Vulkan, Windows/DX12, AMD/Intel) with optional accelerators behind a trait and a portable wgpu/CPU floor — not Mac-only, not CUDA-locked
- ✅ Hit ~10-15fps live-at-depth-cadence on a base M1 and ~30-60fps on RTX-class, with smooth mesh updates via multi-frame fusion even when depth runs below camera rate
- ✅ Stay under the 8GB unified-memory ceiling with sparse voxel hashing, object pools, out-of-core streaming, and dirty-only extraction
- ✅ Ship fully permissive OSS (dual MIT/Apache, cargo-deny-enforced, weights downloaded at first run) and transfer the camera abstraction toward a drone source later

## What this CANNOT do
- ❌ Survey-grade or globally-consistent metric accuracy over multi-minute free-walk trajectories — without IMU/LiDAR the mesh WILL drift, swim, and double on revisit; it is locally-metric, globally-unreliable
- ❌ True absolute metric scale frame-to-frame: monocular learned depth is scale-approximate (10-20% focal-biased) and per-frame affine-inconsistent — Tesla's fleet uses 8 time-synced cameras + massive multi-vehicle training + (historically) radar/odometry that a single-webcam local system has no equivalent of
- ❌ Robust geometry on degenerate motion (hover/slow pan/pure rotation) or textureless surfaces — exactly where parallax and the photometric VO both fail; scale freezes and regions are flagged low-confidence rather than fused confidently
- ❌ Object semantics / occupancy / classification — output is pure geometry, NO semantic labels (by design)
- ❌ Match SOTA learned dense SLAM (MASt3R-SLAM, DROID, DPV-SLAM) accuracy — those need RTX-4090-class GPUs and/or non-commercial weights and do not run on a base M1
- ❌ Real-time on the absolute portable floor (CPU-only tract/candle ~3-8fps) — usable for correctness/CI, not a live experience without a vendor accelerator
- ❌ Drone deployment as a free transfer — rolling shutter, autofocus-varying intrinsics, motion blur, and outdoor/aerial domain shift are an unproven separate milestone, not implied by the indoor pipeline
