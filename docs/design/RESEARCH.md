## Real-time monocular METRIC depth models runnable from Rust cross-platform — model survey, smallest-usable-variant, license, ONNX/Rust-crate path, and the single best portable OSS Rust-runnable front-end.

### [high] WINNER: Depth Anything V2 Metric-Small (ViT-S, 24.8M params) is the best portable, OSS-licensed, Rust-runnable metric-depth front-end — run it via the ort crate (ONNX Runtime).
DAV2-Metric-Small is the only model that is simultaneously (a) genuinely small/fast enough for a base M1, (b) Apache-2.0 on BOTH code and the Small weights, (c) proven to export to ONNX (fabio-sim/Depth-Anything-ONNX, opset 17/18 via TorchDynamo) and run cross-platform via ort. Architecture is DINOv2-S encoder + DPT head. Metric checkpoints are domain-specialized: Indoor (Hypersim, max 20m) and Outdoor (VKITTI, max 80m) — you ship the one matching your scene, or use the relative model + your own scale solver.

- **Notes:** Params 24.8M; ONNX F16 ~49.8MB / F32 ~99.2MB (CoreML-packaged Small). Best Rust path: ort 2.x with CoreML EP (macOS/M1), CUDA/TensorRT EP (NVIDIA), DirectML EP (Windows), CPU fallback everywhere. Apache-2.0 = fully OSS-compatible incl. commercial.
- **Source:** https://huggingface.co/depth-anything/Depth-Anything-V2-Metric-VKITTI-Small ; https://github.com/fabio-sim/Depth-Anything-ONNX

### [medium] Measured latency: DAV2 ViT-S ~13.3ms (~75 fps) on RTX4080 @518x518 ONNX; ~24.6ms on M3 Max CoreML; ~33.9ms iPhone 15 Pro Max. Base M1 estimate ~40-60ms (~16-25 fps).
fabio-sim ONNX benchmarks (RTX4080, inference only, excludes pre/post): ViT-S 13.3ms, ViT-B 29.3ms, ViT-L 83.2ms. HuggingFace CoreML (ANE) Small: 24.58ms M3 Max, 33.9ms iPhone15ProMax, 31.1ms iPhone12ProMax. M1 base has ~1/2 the GPU/ANE throughput of M3 Max, so 40-60ms is a realistic single-frame budget — sufficient for 15-25fps depth, and the project's progressive/multi-frame fusion means depth need not run at full camera framerate.

- **Notes:** RTX4080 ViT-S = high confidence (published). M1-base = INTERPOLATED (no direct published M1 number found) — flagged medium. Use F16 to halve memory + boost ANE/tensor-core throughput. 518x518 is native; lower res (e.g. 392) cuts latency further.
- **Source:** https://github.com/fabio-sim/Depth-Anything-ONNX ; https://www.promptlayer.com/models/coreml-depth-anything-v2-small

### [high] Apple Depth Pro is the strongest TRUE-metric-without-intrinsics model but is LICENSE-BLOCKED for an OSS project: weights are apple-amlr (research-only, no commercial use). EXCLUDE from the real-time path.
Depth Pro produces sharp 2.25MP metric depth with absolute scale and NO camera intrinsics required, ~0.3s on a 'standard GPU'. But the HF weights carry the 'apple-amlr' (Apple Machine Learning Research) license: grants use ONLY for Research Purposes, explicitly excluding 'any commercial exploitation, product development or use in any commercial product or service.' That violates the project's permissive-OSS constraint. Also ~0.3s/frame and a large ViT backbone make it too heavy for a base-M1 real-time loop, and there is no official ONNX export.

- **Notes:** License = hard blocker (flag CC-BY-NC-equivalent / research-only). Community TensorRT ONNX exists (yuvraj108c/ml-depth-pro-tensorrt) but does not change the weights license. Useful only as offline Python ground-truth/eval tooling, never shipped.
- **Source:** https://huggingface.co/apple/DepthPro ; https://github.com/apple/ml-depth-pro ; https://machinelearning.apple.com/research/depth-pro

### [high] UniDepth V2 and Metric3D V2 are technically excellent intrinsics-aware metric models with ONNX paths, but BOTH are non-commercial licensed (CC-BY-NC-4.0 / BSD-non-commercial) — license-blocked for OSS commercial use.
UniDepthV2 (CVPR/arXiv 2502.20110, Feb 2025): universal metric depth, predicts its own camera intrinsics, has community ONNX (ibaiGorordo). License: Creative Commons BY-NC 4.0 → non-commercial. Metric3D V2 (arXiv 2404.15506): zero-shot metric depth + surface normals, ViT-S/L/giant2, official ONNX export script + onnx-community/metric3d-vit-small. License: 2-clause BSD 'for non-commercial usage' → non-commercial. Both flagged as blockers; permissive alternative is DAV2-Metric-Small (Apache-2.0).

- **Notes:** UniDepth predicting intrinsics is a real advantage for a drone/unknown-camera target — keep as a research/eval reference, but cannot ship in a permissive OSS product. Metric3D-vit-small is the closest NC competitor on size.
- **Source:** https://github.com/lpiccinelli-eth/unidepth ; https://github.com/YvanYin/Metric3D ; https://huggingface.co/onnx-community/metric3d-vit-small

### [high] ort (ONNX Runtime bindings) is the mature, recommended Rust inference crate; candle is the pure-Rust fallback and already ships a native Depth Anything V2 example.
ort 2.x wraps Microsoft ONNX Runtime and exposes execution providers as Cargo features: CoreML (macOS/M1, incl. ANE), CUDA (>=12.8)/TensorRT (NVIDIA), DirectML (Windows DX12 GPUs), with automatic CPU fallback and per-operator EP fallback chaining. Prebuilt MS binaries for CUDA/TensorRT/CPU; CoreML/DirectML link the system framework. Candle (huggingface) has a native depth_anything_v2 example and DINOv2/ViT/DPT support, runs CPU/CUDA/Metal — a pure-Rust, no-C++-toolchain path that fits the 'no cmake' constraint cleanly.

- **Notes:** ort = battle-tested, fastest, best accel coverage, but pulls a C++ ONNX Runtime binary (download-binary feature avoids local cmake). candle = pure Rust, idiomatic, lighter build, but you maintain the metric pre/post yourself and Metal/ANE perf trails ort+CoreML. tract = pure-Rust ONNX but slower and weaker on big ViT/DPT graphs — not recommended for the hot path.
- **Source:** https://ort.pyke.io/perf/execution-providers ; https://github.com/pykeio/ort ; https://github.com/huggingface/candle

### [high] DAV2 metric variants bake in a focal-length / domain assumption — a direct caution for the project's hard 'metric scale from one camera' problem.
DAV2 metric models are fine-tuned per-domain (indoor 20m / outdoor 80m) and internally assume a training focal length (~470mm-class); when projecting to a point cloud with a different camera, metric scale can be off ~10-20%, and indoor vs outdoor checkpoints are not interchangeable. The relative DAV2 model needs an external scale source. This means the Rust pipeline must add a scale-correction/calibration step (known camera intrinsics, or SLAM-recovered scale, or a learned scale token) regardless of which DAV2 metric checkpoint is shipped.

- **Notes:** For the DJI drone (outdoor, unknown intrinsics) target, plan to feed camera intrinsics into a scale step, or evaluate UniDepth-style intrinsics prediction in offline tooling. Do NOT assume DAV2 metric output is drift-free absolute scale.
- **Source:** https://github.com/depthanything/depth-anything-v2/blob/main/metric_depth/README.md ; https://learnopencv.com/depth-pro-monocular-metric-depth/

### [medium] Newer option Depth Anything 3 (Nov 2025) has an Apache-2.0 metric model but only at LARGE size (0.35B params) — too heavy for base M1 and no proven ONNX/Rust path yet.
DA3 (ByteDance-Seed, released 2025-11-14) improves monocular metric accuracy over DA2. DA3METRIC-LARGE is Apache-2.0 but 0.35B params; the smaller DA3-Small (0.08B)/Base (0.12B) are the general 'any-view' relative series, not metric. No ONNX export documented (exports glb/ply/npz only), no published fps. Not viable for the 8GB/base-M1 real-time front-end today.

- **Notes:** Watch-list item: if/when a DA3 metric Small with ONNX export appears under Apache-2.0, it would be the natural successor. For now DAV2-Metric-Small remains the shipping choice. ZoeDepth is older (DPT/MiDaS-based, intrinsics-bound, heavier) and superseded by DAV2 — not recommended.
- **Source:** https://github.com/ByteDance-Seed/Depth-Anything-3 ; https://depth-anything-3.github.io/

**Relevant crates:** ort (2.x) — ONNX Runtime bindings; PRIMARY inference path. EPs: coreml (M1/ANE), cuda + tensorrt (NVIDIA), directml (Windows), cpu fallback. Mature, fastest, widest accel coverage. Use 'download-binary' feature to avoid local cmake.; candle + candle-transformers — pure-Rust ML; has a NATIVE depth_anything_v2 example + DINOv2/ViT/DPT. FALLBACK / no-C++-toolchain path. Backends: CPU, cuda, metal. Maturity good for ViT/DPT; you own the metric pre/post.; tract — pure-Rust ONNX runtime; portable but slower on large ViT/DPT graphs. Only for niche no-binary CPU targets, NOT the hot path.; burn — Rust DL framework with wgpu/candle/ndarray backends; promising but ONNX import for big DPT graphs is still immature — research-only for now.; ndarray / half — tensor pre/post-processing and f16 weight handling around ort.; image / fast_image_resize — camera-frame resize to 518x518 (or lower) before inference, SIMD-accelerated.; nokhwa — cross-platform camera capture feeding frames into the depth stage.; wgpu — portable GPU compute for the surrounding fusion/mesh stages; can share the GPU with ort/candle.; pyo3 + maturin — ONLY for optional offline export/eval tooling (Python model->ONNX), never on the real-time path.

**Recommendations:**
- SHIP: Depth Anything V2 Metric-Small (ViT-S, 24.8M params, Apache-2.0) exported to ONNX (opset 17/18, F16), run via the ort crate. It is the only model that satisfies ALL constraints: small, fast (~13ms RTX4080 / ~25ms M3 Max / est. 40-60ms base-M1), Apache-2.0 OSS on code AND Small weights, ONNX-proven, and cross-platform via ort's CoreML/CUDA/DirectML/CPU EPs.
- Define a clean Rust DepthBackend trait with one ort-backed impl that selects EP at runtime (CoreML on macOS, CUDA/TensorRT on NVIDIA, DirectML on Windows, CPU everywhere) — exactly the 'optional accel + portable fallback' pattern the project mandates. Add a second candle-backed impl as the pure-Rust, no-cmake fallback using its native depth_anything_v2 example.
- Treat DAV2 metric output as SCALE-APPROXIMATE, not ground-truth absolute: add an explicit scale/intrinsics-correction stage (use known/estimated camera intrinsics, or recover scale from SLAM). Ship indoor (Hypersim/20m) and outdoor (VKITTI/80m) checkpoints and select per deployment; the DJI drone target uses the outdoor checkpoint + intrinsics-driven scale.
- Budget: run depth at 392-518px, F16, on a dedicated pipeline thread; let it run BELOW camera framerate (e.g. 15-20Hz) and rely on the project's progressive multi-frame fusion to integrate detail — this keeps peak memory and latency within the 8GB/base-M1 ceiling.
- EXCLUDE from the shipping path (license blockers, flag in docs): Apple Depth Pro (apple-amlr, research-only), UniDepth V2 (CC-BY-NC-4.0), Metric3D V2 (BSD non-commercial), DA3-Metric-Large (Apache-2.0 but 0.35B params, no ONNX). Keep Depth Pro / UniDepth as offline Python EVAL/ground-truth tooling only.
- WATCH: a future Depth Anything 3 metric Small under Apache-2.0 with ONNX export would be the upgrade path; re-evaluate when available.

**Gotchas:**
- Apple Depth Pro weights = apple-amlr (research-only, NO commercial use) — a HARD OSS blocker despite being the best true-metric-no-intrinsics model. Easy to mistake for usable because the CODE is permissive; the WEIGHTS are not.
- UniDepth V2 = CC-BY-NC-4.0 and Metric3D V2 = BSD-'non-commercial' — both non-commercial; cannot ship in a permissive OSS product even though both have working ONNX exports.
- DAV2 LICENSE SPLIT: ONLY the Small (ViT-S) weights are Apache-2.0. Base/Large/Giant DAV2 weights are CC-BY-NC-4.0. Picking ViT-S is not just a perf choice — it is the ONLY DAV2 metric tier you can legally ship in OSS.
- DAV2 metric is DOMAIN-SPECIFIC and focal-length-biased: indoor (20m) vs outdoor (80m) checkpoints are NOT interchangeable, and metric scale drifts ~10-20% when the real camera's intrinsics differ from training — you MUST add a scale/intrinsics step. Do not assume drift-free absolute scale (directly hits the project's hard problem #1 and #2).
- M1-base fps is INTERPOLATED (no published base-M1 DAV2 number found) — verify empirically before committing the latency budget; M3 Max ~25ms does NOT transfer 1:1 to M1.
- ort pulls a C++ ONNX Runtime binary — to honor the 'no cmake' constraint use the prebuilt download-binary feature (or system ORT), and verify CoreML/DirectML linking on each target; Microsoft does NOT ship prebuilt CoreML/DirectML binaries, only CUDA/TensorRT/CPU.
- fabio-sim ONNX latencies are INFERENCE-ONLY (exclude camera capture, resize, normalization, and depth->pointcloud post). Real end-to-end per-frame cost on M1 will be higher than the quoted 13-25ms.
- candle's depth_anything_v2 example targets the RELATIVE model; running the METRIC variant in candle means porting the metric head/scaling yourself (ort consumes the official ONNX directly with less work).
- CoreML EP does not accelerate every ONNX op on the ANE — some ops fall back to GPU/CPU, so real ANE utilization (and the implied M1 speed) can be lower than the all-ANE CoreML-packaged number suggests.

## Monocular 6-DoF Pose / Visual Odometry / SLAM (no IMU/LiDAR) for the Rust "Gods Eye" pipeline

### [high] ORB-SLAM3 is GPLv3 — a hard license blocker for a permissive OSS project, and would require C++ FFI.
ORB-SLAM3 (UZ-SLAMLab) is released under GPLv3; the authors offer a separate closed-source commercial license on request. GPLv3 is copyleft and virally incompatible with the project's MIT/Apache permissive-preferred constraint: linking it into the Rust core (via a C++ FFI shim) would force the entire distributed binary under GPLv3. It also depends on DBoW2 (BSD, but the bundled vocabulary) and g2o, and needs a C++ toolchain — friction given the 'no cmake' baseline. Recommendation: AVOID for an open-sourced permissive project unless the whole project accepts GPLv3.

- **Notes:** License: GPLv3 (blocker). Portability: C++ FFI required, needs cmake/C++ toolchain (conflicts with cmake-free M1 box). Maturity: extremely mature algorithm but NO native-Rust port; only hand-rolled FFI. Perf: ~30+ fps mono on desktop CPU, feature-based (ORB), CPU-bound so portable across GPUs but heavy single-thread.
- **Source:** https://github.com/UZ-SLAMLab/ORB_SLAM3

### [high] DROID-SLAM is BSD-3 (permissive, OK) BUT CUDA-locked and 11GB+ VRAM — unusable on the 8GB M1 baseline and non-portable.
DROID-SLAM is BSD-3-Clause (commercially usable, license-clean). However it ships custom CUDA kernels (correlation, dense bundle adjustment) and the README states demos need a GPU with >=11GB memory; training needs >=24GB. It runs ~8 fps on an RTX-3090. There is no Metal/wgpu path and no CPU fallback. On Apple Silicon (no CUDA) and on the 8GB unified-memory ceiling it simply cannot run. Fails portability + lean-memory hard constraints.

- **Notes:** License: BSD-3 (clean). Portability: CUDA-ONLY, no Metal/wgpu/CPU path — FAILS cross-platform. Memory: >=11GB VRAM (FAILS 8GB budget). Perf: ~8 fps RTX-3090. Maturity: research PyTorch+CUDA, no Rust.
- **Source:** https://github.com/princeton-vl/DROID-SLAM

### [high] DPVO is MIT-licensed and the strongest accuracy/efficiency monocular deep-VO, but its accuracy core is custom CUDA kernels (cuda_corr, cuda_ba) — CUDA-locked today.
DPVO (Deep Patch Visual Odometry, NeurIPS 2023, princeton-vl) is MIT licensed (verified LICENSE file: no non-commercial restriction) — fully permissive and the best license of the deep systems. It runs 60–120 fps and 2x–5x real-time on a single RTX-3090 using only ~4GB VRAM (would fit memory budget IF it ran on the GPU). But it relies on hand-written CUDA extensions (cuda_corr correlation operator, cuda_ba bundle adjustment) plus PyTorch; these have no portable wgpu/Metal/CPU equivalent shipped. Porting to Apple Silicon would require reimplementing those kernels (e.g. in wgpu compute) — a significant effort but feasible because the license permits it.

- **Notes:** License: MIT (BEST of deep systems, permissive). Memory: ~4GB VRAM (fits 8GB unified). Perf: 60–120 fps / 2–5x real-time RTX-3090. Portability: CUDA custom kernels = NOT portable as-is; needs wgpu reimpl of corr+BA for Metal/cross-GPU. Maturity: PyTorch+CUDA research code, no Rust binding.
- **Source:** https://github.com/princeton-vl/DPVO

### [high] DPV-SLAM (ECCV 2024, same MIT repo) adds loop closure + global optimization to DPVO and is the most attractive deep target — 2x faster than DROID at 1/3 the VRAM.
DPV-SLAM extends DPVO with a proximity/loop-closure backend and an efficient global optimization (classical pose-graph) backend, directly addressing the project's drift + loop-closure requirements. Reported: 27 fps vs DROID-SLAM's 8 fps; comparable accuracy on EuRoC/TartanAir at twice the speed and a third of the VRAM; large margins better than DROID on KITTI; on EuRoC enabling loop closure drops runtime only 58->42 fps. Ships in the same MIT DPVO repo. Same CUDA-kernel portability caveat as DPVO. This is the reference design to mirror: deep front-end for pose + classical pose-graph backend for drift/loop closure.

- **Notes:** License: MIT. Perf: 27 fps (vs DROID 8 fps), loop closure cheap (58->42 fps). Memory: ~1/3 of DROID (a few GB). Portability: CUDA-locked like DPVO. Drift/loop: HAS mid-term + long-term loop closure backends — directly relevant.
- **Source:** https://www.ecva.net/papers/eccv_2024/papers_ECCV/papers/00272.pdf

### [high] DSO/DSM are classical, no-deep-learning monocular direct VO — permissive-ish, CPU-portable, but suffer irreducible monocular SCALE DRIFT and no metric scale.
DSO (TUM, direct sparse odometry) and DSM (Direct Sparse Mapping) are direct photometric methods, CPU-based (no CUDA), so trivially portable across platforms and light on memory. But as pure monocular systems they cannot recover metric scale and exhibit substantial scale drift (the literature's headline weakness; Stereo-DSO exists precisely to fix this with a baseline). DSO is GPLv3 (blocker); DSM is also restrictively licensed (academic). For Gods Eye the metric-scale problem is already solved upstream by the learned metric-depth model, so DSO's main weakness (no scale) is moot — its photometric tracking math is reusable, but importing the GPL code is not. Better to reimplement the direct-alignment idea natively in Rust using the metric depth.

- **Notes:** License: DSO GPLv3 (blocker), DSM academic/restrictive. Portability: pure CPU, fully cross-platform, low memory — GOOD. Perf: real-time on CPU (~30fps single-thread). Weakness: scale drift + no metric scale (moot once metric depth supplies scale). Maturity: mature C++; no Rust port.
- **Source:** https://cvg.cit.tum.de/research/vslam/dso

### [medium] RECOMMENDED PATH: native-Rust RGB-D-style odometry driven by the learned METRIC depth — solves scale 'for free', is fully portable, and has no license/FFI/CUDA baggage.
Because the pipeline already produces metric depth per frame, the depth turns the monocular problem into an RGB-D problem: pose can be estimated by dense/semi-dense direct alignment (minimize photometric + inverse-depth/geometric error) or sparse-feature PnP+ICP between frames, giving metric 6-DoF directly. This is implementable in pure Rust (glam/nalgebra for SE3 Lie algebra + Gauss-Newton/LM; the depth model already runs via ort/candle on wgpu/CoreML/CUDA). It has NO CUDA lock-in, NO GPL, NO FFI, runs on the M1 in unified memory, and is the only option that cleanly survives transfer to a drone camera (just re-tune intrinsics + depth model). Drift is handled with a keyframe pose-graph backend + loop closure (see crates). This is the design the user's prompt already hypothesized, and the research supports it as the best portable/OSS/native-Rust choice.

- **Notes:** License: all-permissive (your own code + MIT/Apache crates). Portability: native Rust, wgpu/CPU, runs everywhere incl. 8GB M1 — BEST. Perf budget: frame-to-keyframe direct alignment ~5–15ms/frame on CPU/GPU achievable; metric depth dominates latency. Risk: accuracy below DPV-SLAM; mitigate with robust losses + pose-graph backend. Maturity: you build it; primitives exist.
- **Source:** https://www.sciencedirect.com/science/article/abs/pii/S0921889015002213

### [high] The native-Rust VO/SLAM ecosystem is IMMATURE: kornia-rs/kornia-slam is the leading effort but SLAM/VO is not yet implemented (planned, GSoC 2026).
kornia-rs (Apache-2.0) is a pure-Rust low-level 3D CV library (image I/O, processing, 3D ops, ndarray-based; GPU via CubeCL planned). As of 0.1.10 (Nov 2025)/0.1.11 (2026) it explicitly does NOT yet implement SLAM or visual odometry — these are roadmap items, with a Google Summer of Code 2026 project to build a visual-inertial SLAM on a nascent kornia-slam crate. So there is no production-ready native-Rust SLAM to depend on today. Practical implication: you cannot drop in a Rust SLAM crate; you implement VO yourself on top of kornia-rs primitives + nalgebra/glam, optionally a g2o-style backend. Flag: this is the immaturity point — budget engineering time here.

- **Notes:** License: Apache-2.0 (clean). Maturity: ALPHA — SLAM/VO NOT yet implemented (planned GSoC 2026); usable as primitive library only. Portability: native Rust, CubeCL GPU planned. Use for: image ops, features, 3D math — not a turnkey SLAM.
- **Source:** https://github.com/kornia/kornia-rs

### [high] MASt3R-SLAM (CVPR 2025) is the SOTA portable-feeling dense mono SLAM with METRIC scale, but its weights are CC-BY-NC (non-commercial) and it bundles DROID-SLAM CUDA code — license + CUDA blockers.
MASt3R-SLAM gives globally-consistent poses + dense metric geometry at ~15 fps on RTX-4090/3080Ti, with built-in loop closure and second-order global optimization and a metric-scale two-view prior — conceptually close to Gods Eye's goal. BUT the underlying MASt3R model weights are licensed CC-BY-NC (non-commercial) — a hard blocker for permissive OSS — and the backend reuses DROID-SLAM CUDA code (CUDA-locked, needs a strong NVIDIA GPU). Not viable as-is for a portable, permissively-licensed, 8GB project. Useful only as architectural inspiration (pointmap matching + pose-graph + loop closure).

- **Notes:** License: code Apache/CC-BY-SA but MODEL WEIGHTS CC-BY-NC = BLOCKER; backend = DROID-SLAM CUDA. Perf: ~15 fps RTX-4090. Portability: CUDA + big-GPU, NOT for 8GB M1. Relevant as design reference only.
- **Source:** https://github.com/rmurai0610/MASt3R-SLAM

**Relevant crates:** nalgebra (MIT/Apache, mature) — SE3/SO3 Lie groups, Gauss-Newton/Levenberg-Marquardt solves for pose estimation and bundle adjustment math; glam (MIT/Apache, mature) — fast SIMD vec/quat/mat for the hot pose path; pair with nalgebra for the solver; kornia-rs / kornia crates (Apache-2.0, ALPHA) — native-Rust image I/O, feature ops, 3D primitives; kornia-slam exists but SLAM/VO NOT yet implemented (GSoC 2026 roadmap) — use as primitives only; ort (Apache/MIT, mature, ONNX Runtime bindings) — run the learned metric-depth model with CoreML/CUDA/DirectML execution providers + CPU fallback; the depth that makes RGB-D-style VO metric; candle (Apache/MIT, maturing, HF) — alternative pure-Rust ML inference path (Metal/CUDA/CPU) if avoiding the ONNX C++ FFI of ort; wgpu (MIT/Apache, mature) — portable GPU compute (Metal/Vulkan/DX12/GL/WebGPU) for any custom correlation/alignment kernels if you reimplement DPVO-style ops portably; rerun (MIT/Apache, mature) — live trajectory + point/mesh visualization and debugging of the VO/SLAM output (also has a MASt3R-SLAM example); arrsac / sample-consensus / cv crates by Rust-CV (MIT/Apache, semi-maintained) — RANSAC, essential-matrix, PnP building blocks; quality varies, vet before depending; imageproc + image (MIT, mature) — feature extraction / pyramid building for sparse-feature or semi-dense tracking; FFI fallback only: cxx (Apache/MIT) — clean C++ FFI IF you must wrap a C++ backend (e.g. g2o pose-graph); avoid wrapping GPLv3 ORB-SLAM3 in a permissive project

**Recommendations:**
- PRIMARY: Build native-Rust RGB-D-style metric VO on top of the learned metric depth. The depth model supplies metric scale, eliminating monocular VO's #1 problem; estimate frame-to-keyframe 6-DoF by direct photometric + inverse-depth alignment (DSO-style math, reimplemented, not GPL-imported) using nalgebra SE3 + Gauss-Newton. This is the only path that is simultaneously permissive-licensed, CUDA-free, runs on the 8GB M1, and transfers cleanly to a drone camera.
- DRIFT/BACKEND: Add a keyframe pose-graph backend with loop closure. Detect loops with a bag-of-words / learned global descriptor (note DBoW2/3 are C++; consider a small Rust HNSW + learned descriptor instead), then run pose-graph optimization (native Rust LM, or cxx-FFI to g2o/GTSAM — both have permissive-ish licenses; GTSAM is BSD). This directly answers the loop-closure + drift requirement without IMU/LiDAR.
- ACCURACY UPGRADE PATH (optional, gated): Mirror the DPV-SLAM architecture (MIT-licensed: deep patch front-end + classical pose-graph/loop-closure backend) as the high-accuracy mode. Its CUDA kernels (cuda_corr, cuda_ba) must be reimplemented in wgpu to be portable — significant work, so keep it behind the backend trait as an optional NVIDIA/accelerated path with the native-Rust direct VO as the always-available fallback.
- BACKEND TRAIT: Define a `PoseEstimator` trait with implementations: (a) native-Rust direct-metric VO [portable default, ships everywhere], (b) optional wgpu-accelerated patch VO, (c) optional CUDA DPV-SLAM FFI for NVIDIA boxes. Same trait pattern the project already uses for depth/compute — gives optional-accel-with-fallback and no perf cliff.
- AVOID: ORB-SLAM3 (GPLv3 viral blocker + C++/cmake), DROID-SLAM (>=11GB VRAM, CUDA-only — fails 8GB + portability), MASt3R-SLAM weights (CC-BY-NC non-commercial blocker + DROID CUDA backend). Use them only as design references, not dependencies.

**Gotchas:**
- License traps: ORB-SLAM3 = GPLv3 (viral, kills permissive OSS); DSO = GPLv3; MASt3R model weights = CC-BY-NC (non-commercial — blocks the whole project). DPVO/DPV-SLAM (MIT) and DROID-SLAM (BSD-3) are the only license-clean deep options. Audit weights AND code separately — a permissive codebase can ship non-commercial weights (MASt3R).
- CUDA lock-in is the dominant portability blocker: DROID-SLAM, DPVO/DPV-SLAM, and MASt3R-SLAM all ship hand-written CUDA kernels (correlation, bundle adjustment) with NO Metal/wgpu/CPU fallback. On the Apple M1 baseline (no CUDA) none run unmodified. Reimplementing those kernels in wgpu is feasible but non-trivial — do not assume a quick port.
- Memory cliff: DROID-SLAM needs >=11GB VRAM (demos) — physically impossible on the 8GB unified-memory M1. DPVO's ~4GB is the only deep option that could fit, and only if ported off CUDA.
- Native-Rust SLAM does not exist yet: kornia-slam is roadmap/GSoC-2026, not implemented. Rust-CV crates (arrsac, essential-matrix, PnP) are semi-maintained and uneven. You WILL be implementing the VO/backend yourself — budget for it; this is the ecosystem-immaturity flag the prompt asked about.
- Metric scale is solved upstream, not in the VO: do NOT pick a VO for its scale handling — let the learned metric depth own scale and feed it into RGB-D-style alignment. But beware: per-frame depth scale inconsistency (depth model drift between frames) re-introduces scale drift; enforce scale consistency across keyframes in the backend.
- Drone transfer caveats: a DJI Mini camera has rolling shutter, autofocus (changing intrinsics), aggressive motion blur, and fast 6-DoF motion — direct/photometric methods are sensitive to large optical flow and rolling shutter (a known DSO weakness). Plan for online intrinsics/auto-exposure handling and robust losses; validate the depth model generalizes to aerial/outdoor scenes (most metric-depth models are indoor-biased).
- No IMU = no gravity/scale anchor: without IMU or LiDAR, drift correction rests entirely on loop closure + the depth-supplied scale. Pure VO will still drift on long trajectories with no loops — make loop closure first-class, not an afterthought.
- Don't FFI a GPLv3 library into a permissive binary thinking FFI 'isolates' it — linking (even via FFI) still triggers GPLv3 copyleft on the distributed combined work. The MIT/BSD options (DPVO/DROID) or your own native code are the only safe routes.

## Incremental surface fusion to triangle mesh in real time on Rust + wgpu (NOT CUDA-locked): TSDF+Marching Cubes vs surfel fusion vs voxel-hashing, mapped to WGSL compute, with a recommendation for the cheap-incremental + adaptive-LOD goal of Gods Eye on a base M1.

### [high] A portable WebGPU marching-cubes compute pipeline already runs at near-Vulkan speed; this is the reference impl to port to wgpu/WGSL.
Will Usher's WebGPU marching cubes extracts a 256^3 grid (16.77M cells) in ~31-37 ms on an RTX 3080 vs ~27-33 ms native Vulkan -- 'performance on average very close to native Vulkan.' Pipeline is exactly the three WGSL kernels you need: (1) active-voxel detection (per-cell case index), (2) stream compaction via exclusive prefix-scan to build the active-voxel list, (3) vertex output with computed write offsets. This maps 1:1 to wgpu compute (storage buffers + atomics/scan). Because the design only fuses a sparse shell around the camera (not a dense 256^3), a base M1 (8-core GPU, ~2.6 TFLOPS, roughly 1/8 the RTX 3080) extracting only the few thousand active blocks per frame should hit well under a 10-15 ms budget. Port effort is real but the algorithm and shader structure are public.

- **Notes:** Perf: 256^3 in ~31ms on RTX 3080; M1 is ~8x weaker GPU so dense 256^3 would be ~250ms -- which is exactly why you must use voxel-hashing to extract only active blocks, not the full volume. License not stated on the page; verify the GitHub repo license before copying code (treat as model-knowledge until checked). Cross-platform: WebGPU/wgpu so Metal/Vulkan/DX12/GL all covered from one WGSL source.
- **Source:** https://www.willusher.io/graphics/2024/04/22/webgpu-marching-cubes/

### [high] Voxel-hashing is the substrate that makes incremental updates and adaptive resolution cheap; it is the right fusion structure for an 8GB unified-memory M1.
Niessner et al. 'Real-time 3D Reconstruction at Scale using Voxel Hashing' (SIGGRAPH Asia 2013) allocates voxel blocks (typically 8^3 voxels) only in occupied space via a spatial hash table; each occupied hash entry points to a heap-allocated block storing TSDF+weight(+color). This gives O(1) average insert/lookup, no octree traversal, and full GPU parallelism -- ideal for WGSL where you want flat storage buffers and atomics, not pointer-chasing trees. Incremental update = integrate a depth frame only into blocks intersecting the view frustum near the surface; mesh extraction = run marching cubes only on dirty/active blocks. Out-of-core streaming (keep only blocks within a radius sphere of the camera in GPU memory, stream the rest to host) is the standard trick and matters even on M1's shared 8GB and is mandatory for discrete-VRAM portability.

- **Notes:** Memory: a room at 1cm voxels stored densely would be huge, but hashed blocks store only the thin truncated band around surfaces. Rule of thumb for a single room at ~1-2cm: hashed TSDF fits in a few hundred MB to ~1GB depending on truncation width and block size -- well inside an 8GB budget if you keep the active set near the camera. The MrHash 2025 paper measured ~130-160MB for Replica room-scale voxel storage (see other finding), a concrete datapoint. Portable: hashing logic is plain integer math, trivially WGSL. No CUDA dependency.
- **Source:** https://niessnerlab.org/papers/2013/4hashing/niessner2013hashing.pdf

### [high] Variance-adaptive hashed voxels (MrHash, ToG 2025) directly validate the project's adaptive-LOD thesis and give concrete room-scale memory/fps numbers.
'Resolution Where It Counts: Hash-based GPU-Accelerated 3D Reconstruction via Variance-Adaptive Voxel Grids' (arXiv 2511.21459, ACM ToG 2025) stores multiple resolution levels in a flat hash table and chooses per-region voxel size from local TSDF variance (Welford's online algorithm): high-variance regions stay fine, low-variance flat regions merge to coarse voxels. Reports up to 13x speedup and 4x lower memory vs fixed-resolution, 59-69 fps single-res on RGB-D, 35 fps multi-res, and ~130-160 MB voxel storage for Replica room-scale. This is almost exactly the 'wall = 2 triangles, complex region = many triangles' goal, made quantitative -- but the criterion lives in the TSDF/voxel domain (variance), which is cleaner to compute incrementally than mesh-domain decimation.

- **Notes:** Numbers are on RGB-D/LiDAR + a discrete GPU, not monocular on M1, so treat fps as directional not portable-guaranteed. Code is open-source but CUDA/C++ -- algorithm is portable to WGSL (integer hashing + Welford variance are GPU-trivial) but you reimplement, you do not link CUDA. License of their repo not confirmed; the variance-adaptive *idea* is the takeaway, not their binary.
- **Source:** https://arxiv.org/html/2511.21459

### [high] Surfel fusion is the lighter-weight, lower-latency alternative and runs real-time on CPU alone -- attractive for a no-cliff portable baseline, but it does not natively give you triangles.
HKUST 'Real-time Scalable Dense Surfel Mapping' (ICRA 2019, DenseSurfelMapping) fuses sequential depth+intensity+pose into a globally consistent surfel model; fusion costs <6 ms/frame regardless of scene scale and runs real-time on CPU only (demonstrated on KITTI). Surfels (oriented disks) are cheaper to update incrementally than a volumetric grid (no allocation, no marching cubes) and degrade gracefully on weak hardware -- a good portable fallback or a fast preview layer. Cost: the output is a point/disk cloud, so you need a separate triangulation pass (Poisson, ball-pivoting, or screened Poisson) to hit the 'triangle mesh' requirement, and that pass is the expensive, non-incremental part.

- **Notes:** Perf: <6 ms/frame CPU fusion, scale-independent -- excellent for base M1 and for the no-GPU fallback path. License: DenseSurfelMapping repo license not confirmed on the page (academic ICRA code, often GPL/research -- VERIFY before vendoring; reimplement the algorithm rather than copy if license is non-permissive). No mature Rust surfel-fusion crate exists -- this is a from-scratch Rust implementation, the least mature Rust path of the three substrates.
- **Source:** https://wang-kx.github.io/pdf/surfel_fusion.pdf

### [high] The mature Rust mesh-extraction crates are all CPU (surface-nets / marching-cubes / transvoxel); GPU extraction must be hand-written WGSL. This is the main Rust-maturity gap on this axis.
fast-surface-nets (bonsairobo, MIT OR Apache-2.0) does Naive Surface Nets at ~20M tris/sec on one 2.5GHz i7 core, chunk-friendly with seamless chunk boundaries, SIMD via glam -- production-quality but single-thread CPU. transvoxel 2.0.0 (Nov 2025, MIT OR Apache-2.0) implements Lengyel's Transvoxel = marching cubes with transition cells for stitching blocks at DIFFERENT resolutions -- the canonical solution to crack-free adaptive LOD, but it is CPU and self-described 'experimental,' and cannot flip a block's transition-face status without re-extracting. block-mesh-rs (MIT OR Apache-2.0) is greedy/visible-face quads only (no MC, no surface nets), CPU, single-thread, 'no releases published.' There is NO mature crate doing TSDF fusion or GPU marching cubes in wgpu today.

- **Notes:** Licenses all permissive (MIT/Apache) = safe to depend on or fork. Maturity: surface-nets solid, transvoxel experimental, block-mesh stalled. All CPU. Gap: GPU/wgpu fusion+extraction is greenfield Rust -> you write the WGSL kernels yourself (port Will Usher's pipeline). transvoxel is the reference for the crack-free LOD stitching logic even if you reimplement on GPU.
- **Source:** https://lib.rs/crates/transvoxel

### [medium] Concrete latency/threading plan: separate the fast per-frame fusion thread from the slower dirty-block mesh-extraction thread; only the second touches marching cubes.
Pipeline budget for base M1 at ~15-30 fps target: (a) depth+pose arrive; (b) GPU fusion kernel integrates the frame into hashed TSDF blocks in the frustum -- a few hundred microseconds to a couple ms since only near-surface blocks update; (c) mark touched blocks dirty; (d) on a separate cadence (e.g. every N frames or as a budget allows), run the 3-kernel MC pipeline (active detect -> prefix-scan compaction -> vertex emit) ONLY over dirty blocks, double-buffering the vertex buffer so the viewer never stalls. This is the 'compute follows information' design: flat low-variance walls become coarse blocks (few MC cells, ~2 tris), complex regions stay fine. wgpu's single WGSL codebase covers Metal on M1 and Vulkan/DX12 on Linux/Windows; the optional accel backends (CoreML/CUDA) belong to the depth model, NOT this fusion stage -- fusion is pure wgpu everywhere with no perf cliff.

- **Notes:** Latency estimates are model-knowledge extrapolated from the RTX-3080 MC numbers scaled to M1 and the <6ms surfel datapoint; not directly measured on M1. The key portability win: keeping fusion+extraction entirely in wgpu/WGSL means zero CUDA lock-in and identical code on Linux+NVIDIA, Windows, AMD, Apple Silicon. Prefix-scan in WGSL is the one non-trivial primitive -- but it is exactly what Will Usher's reference implements.
- **Source:** model-knowledge

**Relevant crates:** wgpu (latest 0.20+/22+, MIT OR Apache-2.0, mature) -- portable GPU compute, the foundation; one WGSL source runs Metal/Vulkan/DX12/GL/WebGPU; fast-surface-nets (MIT OR Apache-2.0, solid/mature, CPU) -- ~20M tris/s single core, chunk-friendly seamless isosurface extraction; good CPU fallback path; transvoxel 2.0.0 (Nov 2025, MIT OR Apache-2.0, experimental, CPU) -- Lengyel Transvoxel = MC + transition cells for crack-free adaptive-LOD block stitching; reference for LOD logic even if reimplemented on GPU; block-mesh-rs (MIT OR Apache-2.0, stalled/no releases, CPU) -- greedy + visible-face quads only; useful for flat-block coarse meshing, NOT for isosurface; marching-cubes (crates.io, permissive, CPU) -- straightforward CPU MC for prototyping/validation against the GPU port; glam (MIT OR Apache-2.0, mature) -- SIMD vec/mat math used by the meshers and your hashing/integration code; bevy or three-d or rerun (all permissive) -- live viewer for the streamed double-buffered mesh; rerun is purpose-built for incremental 3D logging; NO existing crate for: hashed-TSDF fusion or GPU/wgpu marching cubes -- this is greenfield WGSL you must author (port Will Usher's WebGPU MC pipeline: active-detect -> prefix-scan -> vertex-emit)

**Recommendations:**
- Adopt voxel-hashing + TSDF as the primary fusion substrate: it is the only one of the three that makes BOTH cheap incremental updates (integrate/extract only frustum-near, dirty blocks) AND adaptive resolution (per-block voxel size) natural, and its flat-hash + integer-math design ports cleanly to WGSL storage buffers with no octree pointer-chasing.
- Extract triangles on GPU by porting Will Usher's WebGPU marching-cubes 3-kernel pipeline (active-voxel detect -> exclusive-scan stream compaction -> vertex emit) to wgpu/WGSL, running it ONLY over dirty blocks each cadence -- this keeps a base M1 far under budget since you never extract the full volume.
- Drive adaptive LOD from the TSDF/voxel domain using the MrHash variance criterion (Welford online TSDF variance per block): low-variance => coarse block => ~2 triangles for a wall; high-variance => fine block. This is cleaner and more incremental than mesh-domain decimation and is validated at room scale (~130-160MB, real-time).
- Use transvoxel's transition-cell algorithm as the blueprint for crack-free seams between blocks meshed at different resolutions; reimplement its transition logic in your WGSL extractor (the CPU crate itself is experimental and re-extracts on LOD change, so don't depend on it at runtime).
- Keep a surfel-fusion path (HKUST DenseSurfelMapping algorithm, reimplemented in Rust, <6ms/frame CPU) as the portable no-GPU fallback and fast preview layer; accept that it needs a separate, non-incremental triangulation pass to satisfy the triangle-mesh output.
- Architect as two threads: a fast per-frame wgpu fusion kernel and a slower dirty-block wgpu extraction kernel with a double-buffered vertex buffer to the viewer -- entirely wgpu/WGSL so there is one codebase and no perf cliff across Apple Silicon, Linux+NVIDIA, Windows, and AMD; reserve CoreML/CUDA backends strictly for the depth model, never for fusion.

**Gotchas:**
- No mature Rust crate exists for TSDF fusion OR GPU/wgpu marching cubes -- every GPU extraction and fusion kernel is hand-written WGSL. The CPU mesher crates (surface-nets/transvoxel/block-mesh) won't carry the real-time GPU path; budget real engineering for the WGSL port. This is the single biggest Rust-maturity gap on this axis.
- WGSL exclusive prefix-scan (stream compaction) is the one non-trivial GPU primitive the MC pipeline needs and wgpu has no built-in for it -- you implement it (Will Usher's reference does). Atomics + subgroup ops vary in support across Metal/Vulkan/DX12/GL backends; test the GL/older-backend fallback or you risk a portability cliff.
- Dense extraction is a trap on M1: a full 256^3 marching cubes is ~31ms on an RTX 3080 and would be ~250ms on a base M1 (~8x weaker GPU). The whole design depends on NEVER extracting the full volume -- only dirty/active hashed blocks. If the dirty-set grows unbounded (drift, loop closure re-touching old blocks), latency spikes.
- Crack-free seams between adjacent blocks at different LOD are mandatory and non-trivial -- plain marching cubes on mixed-resolution blocks produces holes. You must implement transition cells (Transvoxel) or constrain neighbor LOD to differ by at most one level; transvoxel the crate cannot change a block's transition status without full re-extraction.
- Memory: hashed TSDF for one room is a few hundred MB to ~1GB at 1-2cm (MrHash measured ~130-160MB on Replica), which fits 8GB -- but ONLY with out-of-core streaming (keep blocks within a radius of the camera on-GPU, evict the rest to host). The design must NOT assume zero-copy unified memory: discrete GPUs need explicit host<->device block streaming, so build the streaming path from day one.
- License verification still pending on two reference impls: Will Usher's WebGPU-MC repo and HKUST DenseSurfelMapping show no license on their landing pages (academic code is often GPL/research-only). Confirm before vendoring; reimplement the algorithm from the paper rather than copy code if the license is non-permissive. The Rust crates (wgpu, fast-surface-nets, transvoxel, block-mesh) are all confirmed MIT OR Apache-2.0 and safe.
- MrHash/variance-adaptive and the Niessner numbers are RGB-D/LiDAR on discrete GPUs -- monocular learned-depth is noisier, so per-block TSDF variance will be inflated by depth-model error, not just true geometric complexity. You may need to subtract an expected depth-noise floor before using variance as the LOD criterion, or coarse walls will wrongly stay high-res.

## Adaptive-LOD / planar / progressive triangle meshing in Rust (RANSAC/region-grow plane detection, QEM simplification, Hoppe progressive meshes + observation-driven refinement, adaptive TSDF/octree, curvature-adaptive remeshing, confidence weighting), with concrete Rust crates, maturity, performance and licenses for Gods Eye.

### [high] meshoptimizer is the single most load-bearing mesh-simplification dependency available to Rust, and it just hit a stable 1.0 (Dec 8 2025), making it the recommended QEM-style simplifier for the adaptive-LOD stage.
meshopt_simplify is a quadric-error-metric (Garland-Heckbert family) edge-collapse simplifier that produces a SECOND index buffer reusing the original vertex buffer (so it is non-destructive / cheap to re-LOD), follows topology to preserve seams/borders, and crucially returns a normalized [0..1] error via the result_error out-param plus accepts a target_error. That return value is exactly what an observation-driven refiner needs: simplify a region to a chosen geometric tolerance and read back how far you actually deviated. Throughput is ~3M triangles/sec single-thread for meshopt_simplify and ~5-6x that (~20M tris/sec) for the topology-ignoring meshopt_simplifySloppy. v1.0 stabilized the C ABI; v1.1 added meshopt_SimplifyRegularize/RegularizeLight (more uniform triangle sizes) and meshopt_SimplifyVertex_Priority (bias individual vertices to survive) on top of the existing SimplifyLockBorder / ErrorAbsolute / Sparse / Prune / Permissive flags.

- **Notes:** Perf: ~3M tris/s (simplify), ~20M tris/s (sloppy), single desktop core; embarrassingly parallel across mesh chunks via rayon. License MIT (permissive, OSS-clean). Cross-platform: pure portable C, no platform deps. Rust path is a thin FFI binding (meshopt crate / meshopt-rs), not native Rust.
- **Source:** https://meshoptimizer.org/ , https://zeux.io/2025/12/09/meshoptimizer-v1/ , https://github.com/zeux/meshoptimizer/releases/tag/v1.1

### [high] meshopt_SimplifyLockBorder is the canonical, free crack-avoidance mechanism for a chunked incremental mesh and should be the backbone of the anti-crack strategy.
The flag 'restricts the simplifier from collapsing edges that are on the border of the mesh ... useful for simplifying mesh subsets independently, so that the LODs can be combined without introducing cracks.' For Gods Eye this means: tile the fused surface (e.g. per TSDF brick or per spatial-hash chunk), lock chunk-boundary edges, simplify each chunk's interior independently and in parallel to its own error budget, and neighboring chunks stay watertight at shared borders. Vertex-priority (v1.1) lets you additionally protect feature vertices (corners where planes meet) so walls collapse but edges survive.

- **Notes:** This is the lowest-risk crack fix because it needs no custom geometry code. Cost: locked borders cap how coarse a wall-spanning-multiple-chunks can get; mitigate with a hierarchical re-chunk (coarsen chunk size as a region stabilizes). License MIT. Maturity: production-grade (used by Bevy, glTF tooling).
- **Source:** https://github.com/zeux/meshoptimizer , https://github.com/zeux/meshoptimizer/releases/tag/v1.1

### [high] For the meshing-from-TSDF stage, the crack-free multi-resolution problem is already solved by the Transvoxel algorithm, and there is a mature permissive Rust crate (transvoxel 2.0.0).
Transvoxel introduces 'transition cells' with a modified marching-cubes lookup table that seamlessly stitches voxel regions meshed at different resolutions, eliminating the cracks/T-junctions that naive marching-cubes produces at LOD boundaries. The Rust crate transvoxel (Gnurfos/transvoxel_rs) is at 2.0.0, MIT OR Apache-2.0, tagged isosurface/marching-cubes/voxel. This is the right primitive if you adopt an adaptive/octree TSDF where different bricks carry different voxel resolution and you need their surfaces to join without seams - an alternative anti-crack path to the meshopt LockBorder approach (and complementary: use Transvoxel for inter-resolution joins, LockBorder for same-resolution chunk independence).

- **Notes:** License MIT/Apache-2.0 (clean). Pure Rust, no C/cmake. Maturity: small but functional crate, single-maintainer; 2.0.0 published ~early 2026. Perf unstated but marching-cubes-class (fast). Caveat: Transvoxel works on a smooth SDF field, so it pairs with a smooth TSDF, not with the planar-primitive path.
- **Source:** https://crates.io/crates/transvoxel , https://github.com/Gnurfos/transvoxel_rs , https://transvoxel.org/

### [high] fast-surface-nets is the recommended default isosurface extractor for the TSDF->mesh step on modest hardware: ~20M triangles/sec/core, pure Rust, permissive.
fast-surface-nets extracts an isosurface from an SDF sampled on a regular grid using Naive Surface Nets (averaged edge crossings rather than Dual Contouring's Hermite data), reported at ~20M triangles/sec on a single 2.5GHz core using small lookup tables and SIMD. Surface Nets produces fewer, better-shaped quads than marching cubes and gives a vertex-per-cell layout that downstream QEM (meshopt) simplifies cleanly. For Gods Eye's base-M1 8GB budget this is the cheap, portable extractor; feed its output into meshopt_simplify with LockBorder per chunk.

- **Notes:** Perf ~20M tris/s/core (author-reported, single i7 core). Pure Rust + SIMD, no C deps, fully cross-platform. License MIT OR Apache-2.0. Maturity: well-known in Rust voxel ecosystem (bonsairobo), stable. Does NOT do crack-free multi-res by itself - that's why you add per-chunk LockBorder or Transvoxel.
- **Source:** https://docs.rs/fast-surface-nets/latest/fast_surface_nets/ , https://crates.io/crates/fast-surface-nets

### [high] There is NO mature, dedicated Rust crate for RANSAC/region-growing PLANE detection or for Hoppe progressive-mesh (edge-collapse/vertex-split with a vsplit log) - these are the two pieces you will most likely write yourself.
Searches surface academic papers (Garland-Heckbert QEM 1997, Hoppe progressive meshes, half-edge+modified-QEM) but no production crate that does plane RANSAC + region growing on point clouds, nor a progressive-mesh refinement log, in idiomatic Rust. The planar-primitive differentiator ('a wall -> 2 triangles') therefore needs custom code: total-least-squares plane fit (SVD via nalgebra), RANSAC sampling, normal-based region growing on the fused points/voxels, then emit a 2-triangle quad for inlier planar regions and fall back to surface-nets+QEM for non-planar. Progressive/observation-driven refinement (coarse-on-arrival, detail accrues over frames) is also custom: maintain per-region confidence/observation-count, and either (a) re-run meshopt with a shrinking target_error as evidence accumulates, or (b) implement a true Hoppe vsplit/ecol record so refinement is incremental rather than re-simplifying from scratch.

- **Notes:** This is the main Rust-maturity GAP for this axis. Mitigation: nalgebra/glam are mature for the linear algebra; RANSAC + region-grow is ~a few hundred lines; progressive-mesh-via-re-simplify (re-call meshopt at finer target_error) avoids writing a half-edge vsplit engine and is the pragmatic real-time choice. No FFI needed beyond meshopt itself.
- **Source:** model-knowledge + https://medium.com/@wanze.xie/mesh-simplification-and-progressive-mesh-implementation-bbe26858db03

### [medium] threecrate advertises exactly the wanted feature set (GPU TSDF + marching cubes + QEM simplification, wgpu) but its maturity is unproven and it should be treated as a reference, not a load-bearing dependency.
threecrate (umbrella + component crates) lists GPU-accelerated TSDF integration via wgpu, marching cubes, and mesh simplification including quadric error metrics, edge collapse, and clustering-based; dual MIT/Apache-2.0. However it ranks only ~#772 in the Math category with no visible download/star traction, version jumped 0.7.1->0.8.0 in spring 2026, and RANSAC/region-growing are NOT listed. The very broad feature surface appearing fully-formed in a low-traction new crate is a maturity red flag - vendor-lock risk is high.

- **Notes:** License clean (MIT/Apache). wgpu-based = portable by construction (Metal/Vulkan/DX12/GL). Risk: untested in production, possibly thin/AI-assisted, missing the plane-detection pieces you most need. Recommendation: mine it for the wgpu-TSDF shader approach but build the pipeline on meshopt + fast-surface-nets + custom planar code rather than depending on threecrate.
- **Source:** https://lib.rs/crates/threecrate , https://crates.io/crates/threecrate

### [high] For the chunked-voxel scaffolding and greedy-quad meshing of flat regions, the bonsairobo ecosystem (block-mesh, building-blocks/ilattice, ndshape) is the mature Rust toolchain.
block-mesh-rs implements greedy meshing (Lysenko) producing ~1/3 the quads of naive face meshing at ~3x cost, plus simple face meshing; it is the de-facto Rust voxel mesher used by bevy_voxel_world and others. Greedy meshing is itself a cheap planar-LOD primitive for axis-aligned flat regions. The ecosystem also documents the standard skirt trick (emit a downward skirt of geometry at coarse-chunk borders) to hide LOD-boundary cracks when you don't want full Transvoxel.

- **Notes:** License MIT OR Apache-2.0. Pure Rust, cross-platform. Maturity: widely used, stable. Relevance caveat: greedy meshing assumes a blocky/voxel surface and axis-aligned planes - good for the coarse scaffold and Manhattan walls, but the smooth/curved fusion path still wants surface-nets+QEM. Skirts are a flicker-free but slightly-overdraw crack hack vs the cleaner LockBorder/Transvoxel.
- **Source:** https://github.com/bonsairobo/block-mesh-rs , https://docs.rs/block-mesh

### [medium] meshopt-rs bindings build WITHOUT cmake (they vendor the C source and compile via the cc crate), which directly satisfies the project's no-cmake / one-command-cargo-build constraint.
The meshopt / meshopt-rs binding vendors meshoptimizer as a git submodule and uses a build.rs (cc crate) to compile the bundled C, rather than invoking an external cmake. So `cargo build` pulls and compiles it with only a C compiler present (clang on the M1, already there for Metal). Binding crate versions tracked through 0.6.2 (Oct 2025) on crates.io; the gwihlidal repo tagged 0.4.0 (Nov 2024) - confirm the published crate wraps meshoptimizer >=1.0 before relying on the v1.1 flags, otherwise pin the submodule yourself.

- **Notes:** No-cmake: satisfied (cc crate). Needs a C compiler (clang/gcc/MSVC) - present on all target platforms, so portability holds. License MIT. Gotcha: verify the binding's vendored meshoptimizer version actually exposes v1.0/v1.1 simplify flags; a pure-Rust port (yzsolt/meshopt-rs) exists but lags the C feature set and is less battle-tested.
- **Source:** https://github.com/gwihlidal/meshopt-rs , https://crates.io/crates/meshopt

**Relevant crates:** meshopt (gwihlidal/meshopt-rs) - FFI binding to meshoptimizer C lib; QEM meshopt_simplify + LockBorder/sloppy/attributes; MIT; builds via cc crate (no cmake); MATURE/production-grade; the load-bearing simplifier; meshoptimizer (C, v1.0 Dec-2025 / v1.1) - upstream; QEM edge-collapse simplifier, ~3M tris/s (simplify) / ~20M (sloppy); MIT; consumed through the binding; fast-surface-nets - SDF/TSDF -> isosurface mesh, ~20M tris/s/core, pure-Rust+SIMD; MIT/Apache; MATURE; default TSDF mesher; transvoxel (Gnurfos/transvoxel_rs) v2.0.0 - crack-free multi-resolution marching-cubes (transition cells); MIT/Apache; pure Rust; SMALL but functional; for inter-LOD seams on a smooth TSDF; block-mesh (bonsairobo) - greedy + simple voxel quad meshing (planar regions -> few quads); MIT/Apache; MATURE; coarse scaffold / Manhattan walls; ilattice + ndshape (bonsairobo) - lattice math + voxel array indexing for the chunked TSDF/octree scaffold; MIT/Apache; MATURE; nalgebra - SVD/total-least-squares for plane fitting in custom RANSAC; Apache-2.0; MATURE; glam - fast SIMD vec/mat math on the real-time path (wgpu-aligned); MIT/Apache; MATURE; threecrate - umbrella claiming GPU-TSDF(wgpu)+marching-cubes+QEM; MIT/Apache; LOW maturity/traction, no RANSAC; reference-only, do NOT depend on; meshopt-rs pure-Rust port (yzsolt) - no-FFI alternative; MIT/Apache; IMMATURE, lags C feature set; fallback only; CUSTOM CODE NEEDED - RANSAC + normal-based region-growing plane detection, primitive (quad) emission, and Hoppe progressive vsplit/ecol log (or the pragmatic 're-simplify at shrinking target_error' substitute): no mature crate exists

**Recommendations:**
- Adopt a two-tier mesher: (1) detect large planar regions with CUSTOM RANSAC + normal-region-growing (nalgebra SVD) on the fused TSDF/points and emit them as 2-triangle quads; (2) extract everything else with fast-surface-nets, then run meshopt_simplify per spatial chunk to an error budget. This is the concrete realization of 'wall -> 2 tris, complex -> many tris.'
- Make anti-crack cost-free by tiling the surface into chunks and calling meshopt_simplify with meshopt_SimplifyLockBorder so each chunk simplifies independently yet stays watertight at shared borders; use meshopt_SimplifyVertex_Priority (v1.1) to protect plane-intersection corner vertices. If you go adaptive/octree TSDF with mixed voxel resolutions, add the transvoxel crate's transition cells to join different-resolution bricks without T-junctions.
- Implement observation-driven progressive refinement WITHOUT a custom half-edge engine: keep a per-region confidence/observation-count, and re-invoke meshopt_simplify with a target_error that SHRINKS as evidence accumulates (coarse on first sight, finer over frames). meshopt returns the achieved error so you can gate refinement on it. This avoids writing a full Hoppe vsplit log while delivering the same coarse-on-arrival behavior; reserve the true vsplit/ecol record for a later optimization if re-simplify CPU cost shows up in profiling.
- Treat meshopt + fast-surface-nets + nalgebra/glam as the load-bearing, mature dependencies; treat threecrate as a reference design for the wgpu-TSDF compute shader only, NOT a dependency, because its maturity/traction is unproven and it lacks plane detection. Budget explicit dev time for the custom RANSAC/region-grow and progressive-refine code - that is the real Rust-ecosystem gap on this axis.
- Verify packaging up front: confirm the published meshopt crate vendors meshoptimizer >=1.0 (so v1.1 flags exist) and that it compiles via the cc crate with only clang present (no cmake) on all three targets; if the binding lags, pin your own meshoptimizer submodule. Keep all isosurface/simplify math CPU-side and rayon-parallel per chunk so it runs identically on Linux/NVIDIA, Windows, and base-M1 with no GPU-feature cliff, reserving wgpu for the TSDF integrate + viewer.

**Gotchas:**
- meshopt_simplify error is normalized to a [0..1] relative scale by default - for METRIC, scale-stable LOD decisions across frames you must use the meshopt_SimplifyErrorAbsolute flag and feed/read errors in world units, or your LOD thresholds drift as the model's bounding box grows during exploration.
- No mature Rust crate does RANSAC/region-growing plane detection OR a Hoppe progressive-mesh vsplit log - both must be hand-written. Do not assume threecrate covers this; it lists TSDF/marching-cubes/QEM but NOT plane segmentation, and its low traction (#772 Math, no download stats, 0.7->0.8 churn in 2026) makes it a vendor-lock risk.
- SimplifyLockBorder prevents collapsing chunk-boundary edges, so a flat wall spanning many chunks can't fully collapse to 2 triangles while borders stay locked - you get a grid of locked seams. Mitigate by hierarchically re-chunking (growing chunk size) once a region's confidence saturates, or by running a global plane-fit pass that bypasses the voxel mesher for confirmed large planes.
- fast-surface-nets and naive marching cubes are NOT crack-free across different resolutions on their own; cracks/T-junctions/flicker at LOD boundaries require an explicit fix - either LockBorder (same-res chunks), Transvoxel transition cells (mixed-res), or coarse-chunk skirts (cheap but adds overdraw). Pick one deliberately; mixing resolutions with no transition logic guarantees visible cracks.
- Flicker/popping comes from re-simplifying a region to a different target each frame as observations jitter. Hysteresis is mandatory: only refine (lower target_error) when observation-count crosses a threshold and never coarsen a stabilized region, otherwise triangles will pop in and out frame-to-frame. The 're-simplify at shrinking error' approach must be monotonic per region.
- The meshopt binding needs a C compiler at build time (it vendors C and uses the cc crate). This satisfies the no-cmake constraint but is NOT zero-toolchain - clang/gcc/MSVC must be present. On the M1 dev box clang ships with the Metal toolchain so it's fine, but document it for Linux/Windows contributors. The pure-Rust port avoids C entirely but lags the v1.x feature set and is less tested.
- WebFetch returned an obviously wrong date ('April 2, 2016') for meshoptimizer v1.1; the authoritative timeline is v1.0 on 2025-12-08 (zeux.io blog) with v1.1 following after, so v1.1 flags (Regularize, Vertex_Priority) are 2026-era. Verify the exact v1.1 release date and that your binding includes it before depending on those flags.

## RUST real-time performance + portable GPU (wgpu compute, ONNX inference runtimes, multi-threaded pipelining, backend trait + optional native accel, per-stage latency budget for ~15-30 fps on base M1)

### [high] wgpu is production-ready for COMPUTE shaders (not just rendering), with a portable backend across Metal/Vulkan/DX12/GL/WebGPU from one WGSL codebase; current stable is v29 (2026), with v24 (2025-01-15) and v25 (2025-04-10) being the relevant 2025 releases.
Compute is a first-class path: storage buffers, dispatch, timestamp queries (web+native, usable for per-pass GPU profiling), and subgroup ops (native-only feature, may differ from WebGPU spec - do not depend on it for the portable default). For this project, the depth post-processing, point-unprojection, TSDF/surfel fusion, and mesh extraction (e.g. marching cubes / planar fitting) are all good wgpu compute candidates. Treat subgroups/mesh-shaders/ray-query as optional fast-paths behind feature flags, never the baseline.

- **Notes:** wgpu v29 current in 2026; Apache-2.0/MIT licensed (fully OSS-permissive). Cross-platform: Metal (Apple), Vulkan (Linux/Android/Windows), DX12 (Windows), GL (fallback), WebGPU (browser). Runs on base M1 via Metal natively. Mature for compute as of 2025-2026; subgroups still native-only.
- **Source:** https://github.com/gfx-rs/wgpu/releases/tag/v25.0.0 , https://www.arewelearningyet.com/gpu-computing/ , https://docs.rs/wgpu/latest/wgpu/struct.Features.html

### [medium] wgpu compute perf vs native is competitive but NOT free: real-world gap is roughly +20% faster to ~1.65x slower depending on backend and workload, dominated by shader runtime not CPU-GPU overhead.
Measured neural-net simulation: wgpu-native v25 was 20% FASTER than a direct Vulkan baseline on NVIDIA A100, but ~20% SLOWER (regression) on H100 v23->v25, and an early Metal/M3 Max result showed wgpu ~1.65x slower than a prior Vulkan impl (later improved ~20% across v23->v25). Implication: budget a ~1.2-1.6x conservative multiplier vs hand-tuned native, profile per-vendor with timestamp queries, and use create_shader_module_unchecked to skip WGSL validation overhead in release builds (validation was a measured Metal regression source).

- **Notes:** Numbers are workload-specific (large NN sim). For our small per-frame compute kernels (depth at ~256-518px, surfel fusion), absolute cost is small; the risk is dispatch/validation overhead on tiny kernels, mitigated by batching passes into one submission.
- **Source:** https://github.com/gfx-rs/wgpu/discussions/6688

### [high] On DISCRETE GPUs (no unified memory) host<->device transfers via staging buffers are the dominant hidden cost; on base M1 unified memory makes this near-zero, but the design must budget transfers so it does not cliff on Linux+NVIDIA/Windows discrete GPUs.
PCIe is orders of magnitude slower than on-GPU bandwidth; wgpu uploads go through a DEVICE_LOCAL|HOST_VISIBLE staging pool (historically capped near 256MB without resizable BAR). Best practice: keep data resident on GPU, chain all compute passes (depth->unproject->fuse->mesh-extract) into a single command-encoder submission, only read back the FINAL mesh/vertex deltas - never intermediate depth or TSDF volumes. On unified-memory M1, Queue::write_buffer can skip staging; on discrete, only the camera frame upload (~1-6MB at 640x480 RGBA) and the small per-frame mesh-delta readback should cross the bus. map_async REQUIRES device.poll() on native or the callback never fires.

- **Notes:** Cross-platform memory: budget ~1-3MB frame upload + small mesh readback per frame on discrete GPUs (sub-millisecond at PCIe3 x16 ~12GB/s). M1 unified: zero-copy possible but do NOT architect around it. Resizable BAR removes the 256MB staging cap on modern discrete desktops.
- **Source:** https://tillcode.com/rust-wgpu-compute-minimal-example-buffer-readback-and-performance-tips/ , https://github.com/gfx-rs/wgpu/issues/3698

### [high] For the metric-depth inference path, ort (ONNX Runtime bindings) is the most mature and fastest Rust option and is the right default; candle is the pure-Rust fallback (no C++ runtime), tract is CPU-only ONNX.
ort 2.0.0-rc.12 is production-ready (not yet API-stable), Apache-2.0/MIT licensed, and exposes runtime Execution Provider selection: CoreML (Apple ANE/GPU, `coreml` feature), CUDA (>=12.8), TensorRT, DirectML (Windows), with CPU as the always-present fallback. EPs register in priority order and silently fall back, which maps directly to the required backend trait. Reported ~3-5x faster than Python ORT and ~4x CPU throughput. candle is HuggingFace's pure-Rust framework (Apache/MIT) - loads HF models, GPU via Metal/CUDA, no C/C++ runtime, ideal where you cannot ship the ONNX Runtime native lib. tract (Sonos, Apache/MIT) is pure-Rust ONNX inference, CPU/NEON only - good as a no-native-dependency fallback but slower and no GPU.

- **Notes:** ort EP fallback chain == the backend trait the project wants: register [CoreML, CUDA, DirectML] then CPU; ORT picks per node. License clean (Apache/MIT) for ort, candle, tract. ort ships ONNX Runtime native lib (also MIT). candle = best pure-Rust portable fallback; tract = CPU-only no-dep fallback.
- **Source:** https://ort.pyke.io/ , https://github.com/pykeio/ort , https://markaicode.com/rust-ml-inference-engines-2025/

Metric3D v2 supports ONNX export but its weights/license must be verified before adoption. Run depth at reduced res (e.g. 256-384px) to cut latency, then upsample.

### [high] Rust multi-threaded pipelining is mature and idiomatic: std::thread per stage connected by crossbeam-channel or flume (bounded, for backpressure + triple-buffering), rayon for data-parallel inner loops, with fixed pools to avoid alloc stalls.
Architecture: one thread per pipeline stage (capture -> depth-inference -> pose/VO -> fusion -> mesh-extract -> viewer), bounded channels of capacity ~2-3 between stages give automatic backpressure and act as the triple-buffer (producer fills slot N+1 while consumer reads N, third slot in flight). crossbeam-channel is the mature default (used inside rayon); flume has very low latency/footprint but is in casual-maintenance; crossfire is a newer ~2x-faster option. Use rayon for embarrassingly-parallel per-pixel/per-surfel work inside a stage. Avoid per-frame heap churn: preallocate frame buffers, depth tensors, and vertex/index pools once and recycle via a bounded channel of reusable buffers (object-pool pattern) so no stage stalls on the allocator.

- **Notes:** All Apache-2.0/MIT. crossbeam-channel = mature default; flume = low-latency but casual maintenance; crossfire = ~2x faster, newer/less proven. rayon mature for data-parallel. Pattern: bounded channels (cap 2-3) = backpressure + triple-buffer; recycle buffers via a return channel to keep peak memory flat (key for 8GB ceiling).
- **Source:** https://leapcell.io/blog/building-robust-concurrent-pipelines-with-crossbeam-and-flume-channels-in-rust , https://lib.rs/crates/crossfire , https://github.com/crossbeam-rs/crossbeam

### [high] Zero-copy camera->GPU: nokhwa is the mature cross-platform capture crate (V4L2/AVFoundation/MediaFoundation) with optional direct wgpu texture output; on M1 unified memory the frame upload is near-free, on discrete it is one small staged upload per frame.
nokhwa (v0.10.x, Apache-2.0/MIT) covers Linux (V4L2), macOS (AVFoundation), Windows (MediaFoundation) from one API and has a wgpu integration feature to push frames toward GPU textures. True zero-copy to GPU is only achievable on unified-memory (M1) or with platform-specific interop; the portable path is a single Queue::write_texture of the ~1-6MB frame per stage, which is sub-millisecond and negligible vs the ~35-55ms depth stage. Color-convert (YUYV/NV12 -> RGB) is a cheap rayon or compute-shader step.

- **Notes:** nokhwa Apache-2.0/MIT, cross-platform (Linux/mac/Win). 'Zero-copy' is realistic only on unified memory; portable default = one small write_texture/write_buffer per frame (sub-ms). Don't architect SLAM around zero-copy since discrete GPUs need the upload.
- **Source:** https://github.com/l1npengtul/nokhwa , https://docs.rs/nokhwa/latest/nokhwa/

### [high] Viewer: egui+wgpu (eframe) or rerun are the mature, portable live-mesh viewers; bevy is a heavier option; three-d/kiss3d/rend3 are lighter. rerun is purpose-built for streaming 3D geometry/point-clouds and is the fastest path to a live mesh viewer.
rerun (Apache-2.0/MIT, built on egui+wgpu) is designed to stream and visualize evolving 3D geometry/point clouds/meshes over time - near-ideal for an incrementally-built mesh and decouples the viewer process from the real-time core (you log mesh deltas, it renders). For an in-process custom viewer, egui+wgpu via eframe is the portable, cross-platform (incl. web) default; render the mesh to a wgpu texture and overlay egui controls. bevy (MIT/Apache) is a full ECS engine - more power, more weight/binary size. three-d, kiss3d, rend3 are lighter wgpu-based renderers. All run on M1 Metal.

- **Notes:** All Apache-2.0/MIT, all wgpu-based -> same portability as core. rerun = fastest to live mesh streaming + process isolation; egui+wgpu/eframe = in-process portable default; bevy = heavier. Viewer GPU work overlaps depth/fusion since it's a separate stage/process.
- **Source:** https://github.com/emilk/egui , https://docs.rs/rend3/latest/rend3/ , https://github.com/sebcrozet/kiss3d

Mature.; candle (Apache/MIT, HuggingFace) - pure-Rust inference fallback, loads safetensors, good WASM; growing maturity.; tract (Apache/MIT) - mature pure-Rust CPU ONNX/NN inference; conservative fallback.; kornia-rs / kornia-3d / kornia-bow / kornia-apriltag (Apache-2.0) - native-Rust CV: depth->3D, SE3 transforms, ICP, bag-of-words loop closure. v0.1, immature, no SLAM yet.; threecrate (MIT/Apache, v0.8) - Rust+wgpu TSDF volume integration, marching cubes, Poisson/BPA/Delaunay, GPU ICP/normals; pre-1.0, API churn but a real head-start for the fusion->mesh stage.; wgpu (MIT/Apache) - portable GPU compute Metal/Vulkan/DX12/GL/WebGPU; host for our custom DBA / TSDF / LOD-mesher compute kernels.; glam or nalgebra (MIT/Apache) - SE3/SO3 math, Jacobians for photometric/dense bundle adjustment (re-authoring DROID/TANDEM BA in Rust).; nokhwa (Apache-2.0) - cross-platform webcam capture for the live input stage.; rerun (MIT/Apache) - live point-cloud/mesh viewer and debugging; alternatively egui+wgpu or bevy/three-d/kiss3d for the live mesh viewer.; argmin (MIT/Apache) - Rust nonlinear optimization, candidate for pose-graph / bundle-adjustment solver backbone.

**Recommendations:**
- BORROW ARCHITECTURE FROM TANDEM (TUM, CoRL'21) as the primary pipeline template: monocular -> learned MVS/metric depth -> sliding-window photometric bundle adjustment for pose -> incremental dense fusion -> live mesh. It is the closest match to the decided Gods Eye pipeline. Re-author entirely in Rust: depth net via ort (ONNX), photometric BA via glam/nalgebra+wgpu, fusion+mesh via threecrate. Do NOT reuse its CUDA/LibTorch code; verify its license before copying any code.
- BORROW THE FUSION->MESH STAGE FROM Open3D (MIT) as the algorithm reference, implemented on top of threecrate's Rust+wgpu TSDF + marching cubes. Use sparse voxel hashing (not dense global TSDF) to fit the 8GB/M1 ceiling, and feed our adaptive-LOD/planar mesher rather than naive uniform marching cubes. MIT license + CPU-and-GPU + cross-platform makes this the safest, most portable building block.
- BORROW THE SLAM BACK-END (loop closure + drift management) FROM RTAB-Map (BSD): its bag-of-words loop-closure + working-memory/graph-optimization strategy directly attacks the no-IMU drift and lean-memory hard problems. Implement in Rust using kornia-bow for BoW and argmin/nalgebra for pose-graph optimization. Use MASt3R-SLAM and DROID-SLAM as CONCEPT references only (two-view metric pointmaps; Dense Bundle Adjustment) - both are CUDA-locked and MASt3R's weights are non-commercial, so neither ships in a permissive cross-platform product.
- FOUND THE WHOLE THING ON kornia-rs (Apache-2.0) + ort + wgpu as the permissive, non-CUDA-locked Rust stack, and accept that the SLAM/DBA, incremental fusion, and adaptive-LOD mesher layers must be built by us (no mature Rust crate exists). Contribute these upstream to kornia-rs where possible.

**Gotchas:**
- LICENSE BLOCKER (model weights): MASt3R/DUSt3R code AND checkpoints are CC-BY-NC-SA-4.0 and require accepting restrictive training-dataset licenses (e.g. mapfree). Incompatible with the project's permissive-OSS 'must run for anyone' rule. The MASt3R architecture is excellent, but you CANNOT ship NAVER weights - you must source/train a permissively-licensed pointmap or metric-depth model.
- LICENSE BLOCKER (depth weights): Depth Anything V2 - ONLY the Small relative model is Apache-2.0. Base/Large/Giant relative models AND ALL metric-depth checkpoints (Indoor/Hypersim and Outdoor/VKITTI, every size) are CC-BY-NC-4.0 (non-commercial). For a permissively-licensed metric-depth path, prefer Metric3D v2 (BSD-2 but '2-clause BSD for non-commercial usage; commercial inquiries to the author' - read carefully) or the Apache-2.0 DA-V2-Small, or train your own metric head. Verify UniDepth's license directly (reported as a Creative Commons variant - likely CC-BY-NC, treat as suspect).
- LICENSE FLAG (OpenMVS): AGPL-3.0 copyleft - usable as Python OFFLINE tooling only; never link it into the distributed Rust core. COLMAP itself is BSD (fine).
- MEMORY BLOCKER for 8GB/M1: DROID-SLAM demo needs >=11GB VRAM, GO-SLAM peaks at 18GB. A global Dense-Bundle-Adjustment / dense-TSDF design simply will not fit. The adaptive-LOD + sparse-voxel + sliding-window design is not optional - it is forced by the memory ceiling.
- PERFORMANCE BLOCKER: MASt3R-SLAM targets an RTX 4090 and GO-SLAM hits only 8 FPS on a 3090. None of the SOTA learned-dense-SLAM systems are real-time on a base M1. Budget for a smaller/quantized depth net (ONNX int8 via ort+CoreML) and frame-to-model tracking rather than full per-frame global BA.
- RUST MATURITY GAP: there is NO mature Rust crate for learned dense SLAM, Dense Bundle Adjustment, or incremental surface fusion. kornia-rs is v0.1 (no SLAM yet), threecrate is v0.8 (pre-1.0, breaking releases). Expect to build the VO/DBA, fusion, and LOD-mesher layers yourself; the ecosystem only gives you primitives (tensors, depth->3D, ICP, TSDF, marching cubes, BoW) and the inference host (ort).
- PORTABILITY CAVEAT: ort delivers the CoreML/CUDA/DirectML+CPU-fallback story cleanly, but it links the C++ ONNX Runtime (a binary dependency that complicates 'cargo build everywhere' and the no-cmake constraint). Keep candle/tract (pure-Rust) as the fallback behind the same inference trait so the project still builds and runs with zero native toolchain.
- SCALE CAVEAT (monocular): RTAB-Map and classic pipelines confirm monocular alone yields NO metric scale. Metric scale must come from the learned metric-depth model (its license is therefore on the critical path) or from a known-size reference / drone telemetry later.
